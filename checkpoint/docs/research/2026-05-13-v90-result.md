# v90 + v90b result — msk_init_hw replay DESTROYS the MAC (2026-05-13)

## TL;DR

Tested two versions of glm's "missing msk_init_hw" hypothesis.  Both bad:

**v90 (full ~25 BAR writes from glm's documented list):**
Kernel HUNG during probe right after "registered ISR on IRQ 1 (MSI)".
No msk_init_hw dmesg lines fired.  Some write in the 0xe80/0xf20/0xf22
range silently wedges the MAC and the kernel.

**v90b (minimal 4 writes: glm's top-3 candidates):**
Kernel survives but **BAR+0x00c = 0 (full clear of MAC_CTRL2) destroys
the MAC.**  All BAR reads return 0 afterwards.  linkreg goes from
0x00000b18 (v89 state) to 0x00000000.  SMI/PHY readability degraded.

```
[5.061] v90b: msk_init_hw DONE: BAR+0x004=0x00000000 BAR+0x00c=0x00000000 BAR+0xe80=0x00000000
[5.428] LP ABILITY (reg5)=0x0000   ← was 0xc5e1 healthy in v89
[5.448] 1000BT_STAT (reg10)=0x4000 ← was 0x3c00 with receivers OK in v89
[6.516] link DOWN (linkreg=0x00000000, BMSR=0x7949)
```

## Hypothesis status

| Agent's v90 hypothesis | Tested | Result |
|---|---|---|
| glm-5.1: msk_init_hw replay (full ~25 writes) | v90 | ❌ kernel hung |
| glm-5.1: msk_init_hw minimal top-3 | v90b | ❌ destroys MAC |
| glm-5.1: BAR+0x00c = 0 (rank #1) | v90b | ❌ this is the destroyer |
| glm-5.1: BAR+0xe80 = 1, 2, 8 (rank #2) | v90b | inconclusive (post-MAC-dead) |
| glm-5.1: BAR+0x004 = 8 (rank #3) | v90b | inconclusive |
| deepseek-v41: BAR+0x030 = 0x10000 RGMII | v90b | ✅ applied, no effect on link |

**Key learning:** what Orbis writes in `msk_init_hw` ≠ what we should
write in `msk_init_hw_replay`.  Orbis's BAR+0x00c = 0 happens at a
specific point in its init flow where it's correct (probably after
master reset and before the MAC powers up).  Replicating it after our
parent_prelude + ISR registration is destructive — those bits in
BAR+0x00c are now load-bearing.

v88's approach (only AND ~0x80, clear bit 7) is the correct one.

## Where we land

All "register tweak" hypotheses from the 4-agent v90 dig are now
exhausted:
- ❌ glm's msk_init_hw replay (destroys MAC)
- ❌ deepseek's BAR+0x030 mode toggle (no effect)
- ❌ kimi: stalled, never produced v90 file
- ❌ gpt5.5: no specific writes proposed

The remaining options:

1. **Revert v90b to v89 baseline.**  v89 was the best state we've had —
   PHY-level handshake completely healthy, just MAC bit 0 not latching.
   Don't keep destructive changes in the patch series.

2. **Phase 3 (real TX/RX descriptor traffic).**  If MAC needs to see
   actual incoming/outgoing packets to declare link UP, we have to
   implement the descriptor format properly and allocate buffers.
   Several hundred LOC but unavoidable.

3. **Try different hardware (cable / partner switch).**  We've been
   testing with one specific cable/switch.  Maybe there's a physical-
   layer incompatibility with this MT7531 silicon that no software
   can fix.

## Aggressive test results from v89 boot (for record)

While we had v89 booted, also tested userspace BAR pokes:

- Writing 0xff to BAR+0x09c (MTS_PKT_ENGINE_CTRL) changed `linkreg
  speed field from bits-set-for-1000M (0xb18) to bits-set-for-100M
  (0xb14)`.  This proved the MAC IS reflecting PHY state changes via
  the speed field, just not the bit 0 link-up flag.
- BAR+0x208 / BAR+0x210 (both = 0x00000001 at POR) accept writes but
  bit 0 clears after writing 0x3.  Not gates.
- 64+ random BAR writes don't crash the kernel (PCIe device absorbs
  invalid writes silently).
- Master reset BAR+0x200 cycle: no effect on link.
- Switch reset BAR+0xf04 cycle: no effect on link.

## v91 plan

**Revert v90b first.**  Drop patch 0010 from series.  Return to v89
state (best baseline: PHY healthy, RF cleared, only MAC bit 0 missing).

Then choose:

**A) Run with v89 indefinitely:** accept that MAC bit 0 doesn't latch
without phase 3, ship "PHY-side ready" status.  Move on to other PS4
Linux work (HDMI, USB are areas with more open issues).

**B) Phase 3 minimal:** allocate proper descriptor rings, send ONE
management frame (ethertype 0xFA42 to switch, per glm v83 finding) via
the TX engine and see if BAR+0x04 bit 0 latches after that.  ~200 LOC.

**C) Hardware variable test:** swap cable, swap switch, see if a
different partner gives different results.  Free, fast, requires user.
