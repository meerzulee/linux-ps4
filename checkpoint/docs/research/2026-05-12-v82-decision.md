# Ethernet bring-up — v82 decision point (2026-05-12, end of session)

After v77 → v81 (six iterations this session), the path forward
splits into two clear options.  This doc captures the choice for the
next session to start from.

## Where we are

Confirmed working (in any path):
  - sky2 / ps4_mts binds the Baikal GBE PCI device 0x104d:0x90d8
  - SMI MDIO C22 protocol at BAR+0x00 — verified RE'd encoding
  - PHY identified as MediaTek MT7531 (id 0x03a29441)
  - Sky2's `phy I/O error` flood eliminated
  - Baikal IRQ block enable (BAR+0x204 = 0x10001388, BAR+0x54 = 0x7bfffe)
    persists across uptime when set explicitly
  - sky2_intr pre-handler W1C-acks BAR+0x50 cleanly

Confirmed NOT working:
  - SMI MDC clock cannot be sustained through sky2's interface-up path
  - Link never reaches UP
  - Live BAR readback shows SMI returning `0x20008000` (DONE bit set,
    data field is residue of last WR_OP bit — MDC not clocking)

## The two paths

### Path A — continue gating sky2's open path (v82..vN)

Identify and gate the next sky2 function that scrambles Baikal MAC
state.  Candidates ordered by likelihood (from v81 negative-result
analysis):
  1. `sky2_mac_init` — GMAC writes at port_base + 0x80 = BAR+0x180-0x200
  2. `sky2_hw_up` — STATUS_LIST_CTRL family at BAR+0xe80+
  3. `sky2_phy_power_up` — PCI config writes via B2_TST_CTRL1

Pros: small incremental patches, builds on v78-v81 infrastructure
Cons: whack-a-mole.  v81 disproved the kimi/glm convergent theory
that B0_IMSK was the killer.  Each gate might fix one symptom and
expose the next.  Kimi estimated ~80% of sky2 open-path code needs
negation for Baikal — we're at ~30% and the SMI still dies.

### Path B — revert sky2 patches, restore standalone ps4_mts.c (recommended)

Revert patches 0008..0011 from `patches/6.x-baikal/0700-network-sky2/`
and the `#ifdef CONFIG_PS4_MTS` removal of Baikal entry.  Restore the
standalone `ps4_mts.c` driver from commit `e2a0eba`.  Then port the
full Orbis `mts_mac_init` sequence verbatim from
`checkpoint/docs/research/2026-05-12-orbis-mts-driver-RE.md`.

Estimated work:
  - ~500 LOC clean from-scratch driver
  - Replays Orbis init exactly (no Yukon assumptions to fight)
  - Phase 1 = link detection (DONE in v77, just needs sustained MDC)
  - Phase 2 = TX path
  - Phase 3 = RX path
  - Phase 4 = ethtool, multicast, etc.

Pros: clean architectural fit.  Sky2's Yukon-2 register-map collisions
are gone.  Orbis's RE'd init sequence is known-correct.
Cons: more LOC than incremental sky2 negation.

## Recommended

**Path B.**  Per kimi's framing: at ~80% negation needed, sky2-as-shell
becomes harder than from-scratch.  The Orbis driver is internally
structured as TWO drivers (msk parent + mts child) precisely because
the per-port mts ifnet has unique register conventions that don't fit
the Yukon-2 msk parent.  Linux equivalent is a single PCI driver but
needs a similar architectural separation in spirit.

## v82 specific plan

1. `git revert` or `git rm` patches 0008..0011 from sky2 series.
   Restore CONFIG_PS4_MTS=y, restore ps4_mts.c via revert of v78's
   sky2 #ifdef change.  Start from the v77 baseline.
2. Add Orbis's full `mts_mac_init` register sequence (RE doc section
   "Register map (BAR0, by offset)" — about 25 writes).  This needs:
   - BAR+0x07c write of 25M (already persists from Orbis, but make
     our driver write it too for cold-boot future-proofing)
   - BAR+0x08 OR 0x07597c00, BAR+0x0c clear bit 7, etc.
   - BAR+0x14/0x18 MAC address bytes byteswapped from SBL value
   - Multicast hash filter setup at BAR+0x1bc..0x1d4
   - BAR+0x30 = 0x10100, BAR+0x74 = 0x2277, BAR+0x78 clear bit 0
   - BAR+0x200 = 0 (master reset clear), BAR+0xac = 9
3. Request_irq + register an ISR mirroring Orbis's `mts_intr`:
   - Enable IRQ block: BAR+0x204 = 0x10001388, BAR+0x54 = 0x7bfffe
   - Walk BAR+0x50 status bits and ack
4. SMI sweep + link poll, watching whether MDC stays alive.

Pass criterion identical to v77 phase-1 spec: dmesg shows real PHY
register values (BMSR 0x7949 etc.) sustained for >5 minutes.

## Open RE work that would unlock v82

Whichever path is chosen, these RE items would help:
  - **`gbe:ctrl` and `gbe:phy_ctrl` kthread function bodies.**
    Resumed in mts_ifup (`FUN_c85ec940`) via softc+0x3150 and +0x31a0.
    Created somewhere we haven't traced (not in attach functions).
    These run continuously and may be what keeps MDC alive in Orbis.
  - **Module-init / SYSINIT for the mts module.**  Probably contains
    the kthread_create calls.  Search the Ghidra binary for the
    DECLARE_MODULE/SYSINIT pattern referencing the gbe driver.
  - **Caller of `baikal_gbe_attach` (FUN_c8511100).**  Reference at
    binary offset 0x71b9d0 (in unallocated_2 segment) — likely a
    device_method_t table.  Need manual data segment analysis to
    follow the pointer chain back to whoever invokes attach.
