# Ethernet bring-up — v82 decision point (2026-05-12, updated post-4-agent dig)

After v77 → v81 (six iterations this session), the path forward
splits into two clear options.  This doc captures the choice for the
next session to start from.

**Updated 2026-05-12 evening:** four independent AI agents (glm-5.1,
gpt-5.5, kimi-k2.6, deepseek-v41) ran Ghidra MCP digs on the Orbis
firmware in parallel.  The results converge into a coherent story
that v81's negative result fits.  See **"Synthesis of 4-agent dig"**
below.  TL;DR: Path B (standalone `ps4_mts.c`) is correct, but the v82
plan needs to add a **periodic SMI heartbeat kthread** as a first-class
requirement, not an afterthought.

## Where we are

Confirmed working (in any path):
  - sky2 / ps4_mts binds the Baikal GBE PCI device 0x104d:0x90d8
  - SMI MDIO C22 protocol at BAR+0x00 — verified RE'd encoding
  - PHY identified as MediaTek MT7531 (id 0x03a29441)
  - Sky2's `phy I/O error` flood eliminated
  - Baikal IRQ block enable (BAR+0x204 = 0x10001388, BAR+0x54 = 0x7bfffe)
    persists across uptime when set explicitly
  - sky2_intr pre-handler W1C-acks BAR+0x50 cleanly

Confirmed NOT working:
  - SMI MDC clock cannot be sustained through sky2's interface-up path
  - Link never reaches UP
  - Live BAR readback shows SMI returning `0x20008000` (DONE bit set,
    data field is residue of last WR_OP bit — MDC not clocking)

## Synthesis of 4-agent dig (2026-05-12)

Four agents ran independently with Ghidra MCP access to the Orbis
firmware.  Three orthogonal root causes were identified, **all real
and all required** for v82 to succeed.

### Finding 1 — Missing kthread heartbeat (glm-5.1 + kimi-k2.6, convergent)

Orbis creates two kthreads at attach time in `FUN_c85ef4b0` (the
PORT_A child attach helper called from `bus_generic_attach`):

  - `gbe:ctrl` = `FUN_c85f0190` (SceGbeMtsCtrl) — IRQ state machine
  - `gbe:phy_ctrl` = `FUN_c85f0480` (SceGbeMtsPhyCtrl) — **periodic
    SMI BMSR reader, 3-second cadence, forever**

The `gbe:phy_ctrl` thread is the **SMI MDC heartbeat**.  Its main loop
wakes on a 3-second `msleep` timeout, reads BMSR via `mts_smi_cl22_read`,
optionally reads MMD vendor pages via `mts_smi_cl45_read`, and on
link-down handles AN restart.  Every iteration exercises the full
SMI handshake (`ARM`, `OP_RD`, poll `DONE`, fetch data).

The SMI clock domain on this chip is **transaction-gated, not
enable-bit-gated**.  Without continuous bus activity (kthread polling),
MDC powers down within ~1 minute and every SMI read returns
`0x20008000` (DONE bit set + residue of WR_OP bit 0x2000).

This is the structural reason for the one-way SMI death seen in
v78..v81.  No combination of register state restores MDC — only
ongoing SMI transactions can.

### Finding 2 — sky2 resets the switch chip on every ifup (deepseek-v41)

BAR+0xf04 on Baikal = **switch chip GPIO reset**.  Orbis `msk_init_hw`
writes it `1`, 12ms delay, `2`, 500ms delay — ONCE during attach.

Linux `sky2_mac_init` (`sky2.c:1002-1003`) writes
`SK_REG(port, GPHY_CTRL) = BAR+0xf04 = GPC_RST_SET (0x01)` then
`GPC_RST_CLR (0x02)` on **every** interface-up, with **no delays**.

That's a switch chip reset without recovery time.  PHYs go silent →
SMI MAC completes transactions locally (DONE bit sets) but MDC never
clocks externally → data field stays at WR_OP residue.

Explains v80 perfectly: the one good BMSR read (`0x7949`) was sky2's
link timer firing in the ~500ms window before the switch reset fully
took hold.

### Finding 3 — Parent prelude order matters (gpt-5.5)

Linux `sky2_reset` matches the Orbis `msk_init_hw` BAR+0x158/0x160
mask values but misses the **prelude choreography** from
`FUN_c85131d0(softc, 1)`:

```
BAR+0xf10 = 1, then BAR+0xf10 = 2
BAR+0xf04 = 1, delay 12ms, BAR+0xf04 = 2  (switch chip reset with timing!)
if chip == 0xbd:
    BAR+0x60 = 0x32100
    BAR+0x64 = 6
    BAR+0x68 = 0x63b9c
    BAR+0x6c = 0x300
BAR+0x120 = 1
BAR+0x11c &= 0xf8ff
BAR+0x158 mode 2, then BAR+0x158 mode 1
```

The 2→1 transition on BAR+0x158 is suspected clock-domain selector
(could be test/calibration latch).  Order is critical because
register-visible final state can look correct while a transition
edge was missed.

### How the three findings combine

| | v80 | v81 | v82 (with all three fixes) |
|---|---|---|---|
| BAR+0xf04 switch reset | ✅ sky2 hits it (no delays) | ✅ still hits | ✅ done ONCE with proper delays |
| B0_IMSK → MAC_CTRL2 bit set | ✅ briefly enables MDC | ❌ gated | n/a (no sky2 in v82) |
| gbe:phy_ctrl kthread heartbeat | ❌ never | ❌ never | ✅ kthread polls BMSR every 3s |
| Parent prelude in order | ❌ partial | ❌ partial | ✅ replayed in mts_probe |
| Result | One BMSR latched, then MDC gates off | Even that one read fails | Sustained BMSR reads, link comes up |

## The two paths

### Path A — continue gating sky2's open path (v82..vN) — REJECTED

Identify and gate the next sky2 function that scrambles Baikal MAC
state.  Even with the BAR+0xf04 gate (deepseek's specific fix), we'd
still need to add a periodic SMI poller separate from sky2's link
timer (because sky2's timer fires too rarely and through write paths
that may re-break MDC).  At that point sky2-as-shell is strictly more
complex than just writing the driver fresh.

Pros: small incremental patches, builds on v78-v81 infrastructure
Cons: whack-a-mole.  v81 disproved the kimi/glm convergent theory
that B0_IMSK was the killer.  Each gate might fix one symptom and
expose the next.  Plus we still need the kthread.

### Path B — revert sky2 patches, restore standalone ps4_mts.c (CHOSEN)

Revert patches 0008..0011 from `patches/6.x-baikal/0700-network-sky2/`
and the `#ifdef CONFIG_PS4_MTS` removal of Baikal entry.  Restore the
standalone `ps4_mts.c` driver from commit `e2a0eba`.  Then port the
full Orbis init sequence verbatim from
`checkpoint/docs/research/2026-05-12-orbis-mts-driver-RE.md` PLUS
the parent prelude and SMI heartbeat kthread from the 4-agent dig.

Estimated work:
  - Phase-1 ps4_mts.c skeleton: already exists (commit `e2a0eba`)
  - Add `mts_parent_prelude()`: ~30 LOC (Finding 3)
  - Add full `mts_mac_init` sequence: ~25 register writes (RE doc)
  - Add `mts_phy_ctrl_thread` kthread: ~50 LOC (Finding 1)
  - Total: ~150 LOC on top of existing phase-1 patch

Pros: clean architectural fit.  Sky2's Yukon-2 register-map collisions
are gone.  Orbis's RE'd init sequence is known-correct.  All three
4-agent findings fold in naturally.
Cons: more LOC than incremental sky2 negation, but each LOC is
RE-grounded.

## Recommended

**Path B** — confirmed.  Per kimi's framing: at ~80% negation needed,
sky2-as-shell becomes harder than from-scratch.  Plus the
kthread-heartbeat requirement (Finding 1) cannot be retrofitted into
sky2 cleanly — sky2's link timer runs in soft IRQ context with
different locking expectations than a dedicated kthread.

## v82 specific plan (updated)

1. `git rm` patches 0008..0011 from sky2 series.  Restore
   `CONFIG_PS4_MTS=y`, restore ps4_mts.c via the existing 0750 patch.
   Start from the v77 baseline + the 4-agent findings.

2. **Add parent prelude** (Finding 3) to `mts_probe` BEFORE
   `mts_mac_init`:
   ```c
   /* Orbis FUN_c85131d0(softc, 1): switch GPIO + clock-domain glue */
   writel(1, base + 0xf10);
   writel(2, base + 0xf10);
   writel(1, base + 0xf04);  msleep(12);
   writel(2, base + 0xf04);  msleep(500);
   writel(0x32100, base + 0x60);
   writel(6,       base + 0x64);
   writel(0x63b9c, base + 0x68);
   writel(0x300,   base + 0x6c);
   writel(1, base + 0x120);
   v = readl(base + 0x11c);  writel(v & 0xf8ff, base + 0x11c);
   v = readl(base + 0x158);  writel((v & ~3) | 2, base + 0x158);
   v = readl(base + 0x158);  writel((v & ~3) | 1, base + 0x158);
   ```

3. **Add full mts_mac_init** (RE doc section "Register map (BAR0, by offset)"):
   ~25 writes — BAR+0x07c = 25M, BAR+0x08 OR 0x07597c00, BAR+0x0c clear bit 7,
   BAR+0x14/0x18 MAC address (byteswapped from SBL), multicast hash filter
   at BAR+0x1bc..0x1d4, BAR+0x30 = 0x10100, BAR+0x74 = 0x2277, BAR+0x78
   clear bit 0, BAR+0x200 = 0 (master reset clear), BAR+0xac = 9.

4. **Request_irq + ISR** mirroring Orbis `mts_intr`:
   - Enable IRQ block: BAR+0x204 = 0x10001388, BAR+0x54 = 0x7bfffe
   - Walk BAR+0x50 status bits and ack

5. **Spawn SMI heartbeat kthread** (Finding 1, CRITICAL):
   ```c
   priv->phy_ctrl_thread = kthread_run(mts_phy_ctrl_fn, priv,
                                        "ps4_mts_phy_ctrl");
   /* Body: while(!kthread_should_stop()) {
    *           u16 bmsr;
    *           mts_smi_cl22_read(priv, priv->phy_addr, MII_BMSR, &bmsr);
    *           msleep_interruptible(3000);
    *       } */
   ```
   Stop via `kthread_stop` in remove.

6. SMI sweep + link poll, watching whether MDC stays alive.

Pass criterion: dmesg shows real PHY register values (BMSR ~0x7949
etc.) sustained for **>5 minutes** (not just one read like v80).
