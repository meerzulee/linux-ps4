## The deeper architectural mistake (just spotted)

Reading 5.4 Aeolia (the working baseline) for compose_msg setup:

```c
static struct irq_chip apcie_msi_controller = {
    ...
    .irq_compose_msi_msg = irq_msi_compose_msg,    // ← KERNEL DEFAULT
    .irq_write_msi_msg = apcie_msi_write_msg,
    ...
};
```

**Aeolia uses the kernel's standard `irq_msi_compose_msg`** which composes
a real x86 LAPIC encoding (addr = `0xFEE0X000 | apic_id<<12`, data = real
vector).  Then `apcie_msi_write_msg` calls `apcie_config_msi(sc, func,
subfunc, msg->address_lo, msg->data)` with the **real LAPIC encoding**.

In v9-v11 we used a custom composer (`bpcie_irq_msi_compose_msg`) that
writes `addr=0xFEE00000 + data=irq_map_index`.  We've been programming
the southbridge's MSI block with **garbage** (irq_map indexes 0..33
instead of LAPIC vectors) for 3 iterations.

Confirming evidence: NO spurious interrupts in any v9-v11 boot.  If
data=0..31 were going to LAPIC directly, vector 0 (=#DE divide error)
would crash.  The southbridge IS intercepting and silently swallowing
the unintelligible values.

## v12 fix (one-line essence)

```c
-    .irq_compose_msi_msg = bpcie_irq_msi_compose_msg, /* Option E v9 — Baikal-magic */
+    .irq_compose_msi_msg = x86_vector_msi_compose_msg, /* v12: real LAPIC encoding */
```

After v12:
- Device MSI cap gets real LAPIC encoding
- `bpcie_msi_write_msg` calls `bpcie_config_msi(sc, func, subfunc, real_addr, real_data)`
- Southbridge programs its per-func block with REAL values
- When device fires MSI, southbridge re-emits with REAL LAPIC encoding
  → fires bpcie's vector → kernel runs `bpcie_handle_edge_irq` → demuxes

The `irq_map[]` field and `bpcie_irq_msi_compose_msg` function become
dead code — can be removed in v13 cleanup if v12 works.

## Lessons captured

- The 5.4 baseline source is the GROUND TRUTH for what works on this
  hardware.  Always cross-check against it before introducing
  architectural innovations.  v9's "BaikalLove composer" innovation
  was an unforced error — Aeolia (the only working sibling) does it
  the standard way.
- "Custom composer" was lifted from feeRnt's `BaikalLove` branch
  (research/upstream-bpcie-survey/feeRnt_x_exp__6.15.4-BaikalLove.c).
  That branch is feeRnt's experimental work; not a confirmed working
  baseline.  We mistook research notes for proven implementation.

## Bottom line

v11 was a clean bug fix on top of v10 (extract func from PCI device,
not from leaf hwirq).  But the BIGGER architectural mistake was in
v9 — using a custom composer instead of the kernel default.  v12
reverts that, completing the Aeolia-faithful port.
