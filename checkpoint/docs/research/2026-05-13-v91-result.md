# v91 result — 🎉 LINK UP at 1000 Mbps Full Duplex (2026-05-13)

## TL;DR

After **12 iterations** (v77 phase-1 skeleton → v91), the PS4 Baikal MTS
ethernet **MAC's `BAR+0x04` bit 0 latched for the first time** and the
host RTL8153 USB-Ethernet adapter sees the PS4 PHY at 1000Mb/s Full.

PS4 dmesg:
```
ISR: link-change IRQ (irq_status=0x00040084, linkreg=0x00000b19)
link UP — 1000 Mbps full-duplex (linkreg=0x00000b19, BMSR=0x796d)
phy_ctrl: link UP (linkreg=0x00000b19)
```

Host RTL8153 `ethtool`:
```
Speed: 1000Mb/s
Duplex: Full
Link partner advertised link modes: 10/100/1000baseT
Auto-negotiation: on
```

Link has been stable for 2+ minutes, no flapping (4 transitions total
in dmesg = 2 DOWN + 2 UP during AN convergence, then steady).

## What was in v91

- (a) Minimal status unit enable: `BAR+0xe80 = 1/2/8` — **didn't stick**
  (`readl(BAR+0xe80)` after writes = 0x00000000; that register doesn't
  exist on Baikal).
- (b) MMD 1 (PMA) + MMD 7 (AN via C45) readback — **all zeros**, so
  those MMDs are unimplemented on MT7531.

So neither documented v91 fix is "the" cause of link UP.

## What likely IS the cause

Three candidates, none cleanly bisected:

1. **Partner change**: previous test partner (some switch/router port)
   was incompatible with our PHY config; RTL8153 USB-Eth on host
   negotiates cleanly with the same software stack. v89 had Phase 2
   cable swap *during* v89's runtime but no full reboot — by v91 both
   the partner change AND a full reboot had happened.
2. **v91's added SMI traffic** — six C45 reads + three BAR+0xe80 writes
   added ~10ms of activity right after `engines started`. May have
   acted as an unintended delay that let AN converge.
3. **kthread AN-restart timing** — `phy_ctrl: AN restart (iter=0, BMCR
   <- 0x1200, reason=first)` fired AFTER engines were already running.
   v89 had this code too but the timing must have differed.

## Evidence the cause is sw, not hw

When the PS4 booted to Orbis (its native OS) between v89 and v91, the
RTL8153 link came up at 1000Mb/s instantly. That proves the PHY
silicon itself works fine with this partner — the Linux driver init
sequence was what kept it gated until now.

## What's missing (Phase 3 = next task)

The ps4_mts driver is intentionally **phase-1 only**: hardware bring-
up, no `net_device`.  The header comment says:

```
* soft-reset + AN restart, and a 1Hz link-status print.  No net_device, no DMA
```

`ip link` on PS4: only `lo, wlan0, ap0, ap1`.  There is NO `mts0` /
`eth0` interface yet.  The link is electrically up but unreachable
from userspace.

To make it useful we need:

- `alloc_etherdev(struct mts_priv)` + `register_netdev` so it appears
  as `mts0` in `ip link`.
- `ndo_open` — kick TX/RX engines on `ip link set up`.
- `ndo_stop` — gate IRQ + stop engines on `ip link set down`.
- `ndo_start_xmit` — pack skb into TX descriptor, kick TX.
- NAPI poll callback — drain RX ring, fill SKBs, `netif_receive_skb`.
- `ndo_get_stats64` — counters from BAR+0x118/0x128/0x12c.
- MAC address: either pull from PCIe EEPROM or generate locally
  admin (`eth_hw_addr_random`).

Descriptor format from v91 research (hints, hermes confirming
exactly in v93):

```
+0x00 u32 status
       bit31  OWN     (1 = HW owns, 0 = driver owns)
       bit30  WRAP    (last entry in ring)
       bit17  RX_VALID (RX-only flag, packet good)
       bits18-19 csum status (3 = IP/TCP OK)
       bits0-10  length (mask 0x7FF, max 2047)
+0x04 u32 buf_addr_lo  (32-bit DMA)
+0x08 u16 vlan_or_csum_start
+0x0a u16 reserved
+0x0c u32 reserved/extension
```

Stride 16 bytes, 4 KB ring = 256 descriptors per direction. Estimated
~200-400 LOC.  Tracked as task #40 (Phase 3).

## Lesson — sanity-check rule paid off again

My self-Explore agent identified MMD 0x1e reg 0x144 bit 5 TXEN_DIG as
a smoking-gun candidate, verified at mainline `mtk-ge-soc.c:976-978`.
Built v92 to clear that bit.  But link came up on v91 before v92
boot — turns out the cable swap + fresh boot was enough.  The TXEN_
DIG patch remains in the tree as a defensive measure but its
necessity is now uncertain — could be tested by intentionally booting
with a known-incompatible partner and toggling the patch.
