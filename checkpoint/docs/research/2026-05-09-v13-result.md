## What v13 didn't fix

`call_irq_handler: 0.227 No irq handler for vector` STILL fires at
t=5.86s — same exact location, same vector.  v13's per-subfunc mask
addition didn't redirect this.

After re-reading:

- The vector 0xE3 (227 dec) is NOT one we ever programmed via
  `bpcie_config_msi`.  Our writes go either `data=0xef` (placeholder
  during alloc) or real LAPIC values like `0x20`.
- Most likely, vector 0xE3 is a one-shot stale fire from orbis-
  firmware-programmed state, between our placeholder write and the
  real vector reprogram.  Not a v13 bug — was already present.
- The actual ICC failure (15 timeouts) is unrelated to vector 0xE3 —
  it's that the ICC mailbox protocol times out, suggesting the chip
  side of ICC isn't responding at all.

Possible deeper cause: `icc.spm_base = pci_resource_start(mem_dev, 5)
+ BPCIE_SPM_ICC_BASE` — bpcie_icc_init relies on Function 6 (MEM)'s
BAR5 being mapped.  If Baikal's MEM function isn't initialized
correctly under our MSI flow, the SPM mailbox region might be
inaccessible.

## What v13 DID prove (positive)

- Boot got further: `/init` ran at t=239s
- ALSA HD-Audio device registered (audio path exists)
- Total boot log doubled in length — more userspace activity
