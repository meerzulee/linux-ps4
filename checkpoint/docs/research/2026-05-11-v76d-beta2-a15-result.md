| Iter | Patch effect | Fault | VCPU |
|---|---|---|---|
| v76d-α | VC4 range = 32 GB | `0x300000` PROTECTIONS=0x02 client=UMC ×10 | stuck |
| v76d-β-1 | GART = 16 GB | `0x300000` (identical) | stuck |
| v76d-β-2-A | flat walk + bind 0x300000000 | `0` PROTECTIONS=0x0c client=UVD ×1 | stuck |
| **v76d-β-2-A1.5** | **+ bind VA 0** | **NONE** | **silent stall** |

Each iteration narrowed the failure mode further. With both 0x300000000 and
VA 0 bound to the same dummy page, the firmware:
- Reads at `0x300000000` → gets zeros (dummy page)
- Treats result as a pointer → null
- Dereferences null → reads at VA 0 → gets zeros (dummy page)
- ... continues to read zeros, never finds valid microcode/data, stalls

## The firmware needs REAL content at 0x300000000

The Ghidra-derived layout was confirmed: **Sony's UVD region 1 (firmware
target, 1920 KB) is mapped at GPU virtual 0x300000000 in vmid 4.** The
firmware executes by fetching from this region, NOT from the VRAM-resident
copy at fw_gpu_addr=0xF00400000.

In mainline amdgpu, UVD firmware lives in a VRAM BO at fw_gpu_addr.
mmUVD_VCPU_CACHE_OFFSET is set to that VRAM address, telling the VCPU
where to find its code. But Sony's firmware build apparently ALSO reads
from a fixed VA (0x300000000) — possibly for a specific data table or
constant, possibly for the entire firmware text in a different access
mode.

With our dummy-zeros at 0x300000000, the firmware:
- Reads "instructions" that are all zeros (NOPs in some ISAs, illegal
  in others)
- Or reads a "config table" filled with zeros (no valid handlers)
- Either way → no meaningful execution → STATUS bit 1 never sets → 2-sec
  timeout

## Next: v76d-β-2-A2 (the planned one)

**Allocate a GTT BO of size ≥ 1920 KB, copy the UVD firmware contents
into the start, bind it at GART virtual 0x300000000.**

Implementation outline:
1. In `uvd_v4_2_start_liverpool` (or earlier in the bring-up):
   - Allocate `amdgpu_bo_create_kernel(adev, size=0x1E0000, AMDGPU_GEM_DOMAIN_GTT)`
   - kmap the BO and `memcpy(kva, adev->uvd.fw->data, fw_size)`
   - The BO's TTM `ttm->dma_address` array has the per-page DMA addresses
   - `amdgpu_gart_bind(adev, 0x300000000ULL, 480, dma_addrs, AMDGPU_PTE_VALID|AMDGPU_PTE_SYSTEM|AMDGPU_PTE_READABLE)`
2. Replace the existing dummy bind at 0x300000000 with the new fw-content
   bind (keep the VA 0 dummy in place as a safety net)

Expected outcome:
- ✅ Firmware reads its own text/data at 0x300000000 → executes properly
  → STATUS bit 1 sets → VCPU runs → ring test passes → UVD works
- 🟡 Firmware progresses further but stalls at a NEW spot (e.g., DMA
  cache invalid, missing message queue init, etc.) — still progress
- ❌ Identical stall (no fault, STATUS=0x4) — the firmware needs more
  than just region 1 mapped; we'd need to also map regions 2 and 3
  (heap/IB at 0x3001E0000, message queue at 0x300304000)
