# v83 hardware result — AN-restart works, link still DOWN, deepseek finds the real blocker (2026-05-12)

## TL;DR

v83 added AN restart to the phy_ctrl kthread per Orbis FUN_c85f0480 event 0x1.
Hardware test confirms:

- ✅ Kthread fires `mts_phy_an_restart` every 15s while link is down
- ✅ Each restart writes BMCR = 0x1200 (AN enable + AN restart)
- ✅ SMI MDC sustained well past 10 minutes (v82's 5-min pass criterion crushed)
- ❌ Link STILL never reaches UP

After ~16 minutes the PS4 SSH stopped responding — almost certainly due
to **unhandled IRQ vector 0x26 flooding** the kernel (`call_irq_handler: 2.38
No irq handler for vector` in dmesg).  We enable BAR+0x204 = 0x10001388 (the
full MAC IRQ block) but **never register an ISR**, so the MAC keeps asserting
IRQs that pile up.

A subsequent 4-agent Ghidra dig (glm-5.1, gpt-5.5, kimi-k2.6, deepseek-v41)
identified the actual blocker for link bring-up.

## The actual blocker (deepseek-v41 finding)

After v83's negative result, four agents ran Ghidra MCP digs.  Three
proposed MMD/C45 additions to mts_mac_init.  But deepseek-v41 found the
real blocker:

> Orbis starts MAC RX/TX engines via `BAR+0x34 |= 1` (RX_RESTART) and
> `BAR+0x38 |= 1` (TX_START) at the end of `mts_init_rings_kick` (called
> from `mts_ifup`).  **Without the RX engine running, the MAC does not
> monitor for link state changes** — BAR+0x04 bit 0 (link up) never gets
> set by hardware, even when the PHY completes auto-negotiation.

`mts_init_rings_kick` (FUN_c85ef1b0) writes:

```
BAR+0x44 = TX_desc_ring_DMA_addr_hi
BAR+0x3c = TX_desc_ring_DMA_addr_lo
BAR+0x48 = RX_desc_ring_DMA_addr_hi
BAR+0x40 = RX_desc_ring_DMA_addr_lo
BAR+0x34 |= 1     ← RX ENGINE START
BAR+0x38 |= 1     ← TX ENGINE START
BAR+0x54 = IRQ_mask_shadow
```

The link bring-up state machine deepseek traced from Orbis:

1. PHY auto-negotiates with partner → AN complete
2. PHY asserts MII link signal
3. **Baikal MAC RX engine (BAR+0x34 bit 0) must be running to latch link**
4. Once latched, BAR+0x04 bit 0 = 1
5. MAC raises BAR+0x50 bit 0x4 (link change IRQ)
6. mts_intr sees it → calls mts_link_change
7. mts_link_change reads BAR+0x04, decodes speed/duplex

Steps 3 is what v82/v83 are missing.  The PHY does step 1-2 correctly
(BMSR shows AN complete, partner advertising), but MAC has no engine
running to relay the signal to BAR+0x04.

## What the other three agents found (corroborating but less load-bearing)

- **glm-5.1**: traced all 77 register writes in `mts_mac_init`, found
  `C45 write MMD 7 reg 0x3c = 0` (might be MT7531 switch SW_RESET), plus
  `BAR+0x04 R/M/W val & 0x7FFFCFFF` (clears speed/control bits) done
  TWICE in Orbis.
- **kimi-k2.6**: ranked AN restart at end of mts_mac_init as #1 (we
  already do this in the kthread, so this is partial confirmation),
  noted Orbis runs mts_mac_init TWICE (at attach + ifup).
- **gpt-5.5**: documented 5 unconditional MMD writes at end of mac_init:
  - `0x189001e = 0x110`
  - `0x122001e = 0xffff`
  - `0x268001f = 0x07f4`
  - `0x3c0007 = 0`
  - `0x33001e clear bit 0x1000`

deepseek's analysis says these MMD writes are mostly Realtek-specific
no-ops on MT7531 (efuse-gated block fails the `efuse(0x6c) & 0x80800000`
check on Sony hardware).  The unconditional tail writes target registers
that "don't exist on MT7531 and are silently ignored."

## Open questions

1. Can we start the RX engine WITHOUT a valid DMA ring?  Or will it
   crash trying to DMA to garbage?
2. If we need rings, we're effectively doing phase 3 (TX/RX paths).
   Is there a "ring of size 0" or "engine quiesced but enabled" mode?
3. The "vector 0x26 has no handler" floods suggest we need an ISR before
   enabling BAR+0x204.  Either register a dummy ISR, OR drop BAR+0x54
   mask to only IRQs we care about (link change = bit 2 = mask 0x4).

## v84 plan

Two competing approaches:

### Option A: Cheap test of deepseek hypothesis

Add to `mts_mac_init`:
```c
writel(readl(bar + 0x34) | 1, bar + 0x34);  // RX engine start
writel(readl(bar + 0x38) | 1, bar + 0x38);  // TX engine start
```

Without DMA rings, MAC might:
- Hang trying to fetch descriptors from `BAR+0x3c..0x48` (all zeros = NULL)
- Or just not transmit anything but still latch link state
- Or assert IRQs that flood without an ISR

**Risk: kernel hang.**  But the diagnostic value is huge — if link
suddenly comes up just from setting these two bits, deepseek's
hypothesis is fully confirmed and we know phase 3 is mandatory for
link-up.

### Option B: Register a minimal ISR first, then try Option A

Add `request_irq(pdev->irq, mts_intr, IRQF_SHARED, "ps4_mts", mts)`
with a stub ISR that just W1C-acks BAR+0x50.  Avoids the IRQ-flood
crash.  Then add the RX/TX engine starts.  Slightly more LOC but
safer to leave running unattended.

### Option C: Skip directly to phase 3 (full TX/RX rings + netdev)

Most work but most likely to actually bring up the link.  Plus needed
eventually anyway.

**Recommendation:** Option B for v84.  Stub ISR + RX/TX engine start.
If link comes up → confirmed, plan phase 3 from a position of
strength.  If link still doesn't come up → narrow remaining suspects
to the C45 MMD writes (option A would have been ambiguous about IRQ
flooding messing with link bring-up).
