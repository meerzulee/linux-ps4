# v89 hardware result — RGMII delay applied, link still won't latch (2026-05-13)

## TL;DR

v89 successfully wrote MT7531 RGMII TX delay (mainline mt7531_phy_config_init
value 0x0404 for both reg 0x13 and reg 0x14), but it did NOT bring up link.

```
v89(g): TX_DELAY r0x13: 0x0044 -> 0x0444 (PAIR_B=4 PAIR_D=4)
v89(g): TX_DELAY r0x14: 0x0044 -> 0x0444 (PAIR_B=4 PAIR_D=4)
```

PHY-level handshake still healthy (RF=0, MS_FAULT=0, both receivers OK,
partner advertising 0xc5e1) — same as v88.  But `linkreg = 0x00000b18`
still — bit 0 doesn't latch.

Additional userspace experiment: forced 100M full duplex no-AN
(BMCR = 0x2100).  Successfully wrote BMCR.  Did NOT bring up link.

## What we've now ruled out as the cause

The MAC's `BAR+0x04 bit 0` does NOT latch despite:

1. ✅ Orbis "unconditional tail" replayed exactly (v87)
2. ✅ MT7531 mainline DSP corrections applied (v88: near-echo, ADC bias,
   MSE threshold, SlvDPSready overwritten to 0x5e, EN_DOWNSHIFT)
3. ✅ Stop advertising 1000BT (v88f)
4. ✅ RGMII TX delay = 0x4/0x4 mainline value (v89)
5. ✅ Force 100M full duplex no-AN (BMCR = 0x2100)
6. ✅ Partner reg 5 RF=0, ACK=1, advertising
7. ✅ Reg 10 MS_FAULT=0, L_RX_OK=1, R_RX_OK=1

The PHY is doing absolutely everything right.  Partner accepts us.
Both receivers report OK.  Link cannot be MORE established at the
PHY/wire level than it already is.

## Conclusion: it's a MAC-side problem we can't fix from userspace

Aggressive experiments confirm:

- `BAR+0x04` bit 0 is purely hardware-driven; userspace writes
  rejected (puts MAC in degraded state)
- 64+ random BAR writes across the entire 4KB BAR don't crash the
  kernel (PCIe device absorbs invalid writes silently)
- Force-speed BMCR=0x2100 doesn't change MAC state
- Master reset BAR+0x200 cycle: no effect
- Switch reset BAR+0xf04 cycle: no effect
- BAR+0x158 mode latch: no effect on link

The MAC simply isn't latching the PHY's link signal.  Possible causes
that we CAN'T test from userspace:

1. **Specific PCS/PMA bit we haven't discovered** — somewhere in the
   undocumented BAR space (we found values at 0x100, 0x110, 0x118,
   0x128, 0x12c, 0x208, 0x210 we never wrote — could be hints)
2. **MAC requires actual data traffic** to declare link — chicken-egg
   where link needs traffic but no traffic without netdev
3. **PHY's MII LINK pin signal** is electrically not making it to MAC
   (a hardware-level issue, not solvable in software)

| 0x118 | 0x000000a7 | unknown |

| 0x1d4 | 0x00000001 | unknown |

BAR+0x208 and BAR+0x210 both = 0x00000001 (only bit 0 set, untouched
by our driver) is intriguing.  These might be:
- "TX engine running" / "RX engine running" status mirror?
- IRQ block sub-enable indicators?
- Some MAC enable bits that need to be set for link to latch?

Worth Ghidra investigation in v90/v91.

## v90+ options

### Option A: Phase 3 (TX/RX descriptor management)

Implement real ring management and try to send a packet.  If MAC's
link bit needs to see traffic to latch, this would fix it.  Several
hundred LOC.  Probably necessary eventually anyway.

### Option B: Ghidra dig for "MAC enable" registers

Ask agents (deepseek/glm/kimi/hermes) what BAR+0x080, 0x098, 0x09c,
0x0b0, 0x0b4, 0x208, 0x210 do in Orbis.  Specifically: does Orbis
write to them before link comes up?  Are there bits that gate the
MAC's link detector?

### Option C: Test on different cable / different partner

Hardware variable.  Our partner might just have an incompatible
implementation with MT7531-via-Baikal.  Try a switch we know works
with PS4 Orbis.

### Option D: Accept v88 as "PHY-level link working" and move on

We've cleared the partner's Remote Fault flag — a major achievement.
PHY is healthy.  If the MAC is just permanently broken (PCB-level
issue), no software fix will help.  Ship v88 as "PHY ready, MAC needs
hardware verification" and move on to other parts of the PS4 Linux
port.
