# v90 research prompt — why MAC's BAR+0x04 bit 0 won't latch

## Context (verified on hardware in v88+v89)

PHY-level handshake is now COMPLETELY HEALTHY:
- BMSR = 0x7969 (AN complete bit 5 set)
- reg 5 (LP ABILITY) = 0xc5e1 → **RF=0** (no Remote Fault), ACK=1
- reg 10 (1000BT_STAT) = 0x3c00 → MS_FAULT=0, L_RX_OK=1, R_RX_OK=1
- Partner ACKs us in AN, both receivers report OK

v88 fixed the SlvDPSready bombshell (Orbis accidentally wrote 0x0c, we
overwrote to 0x5e per mainline mt7531_phy_config_init).
v89 added mainline RGMII TX delay (MMD 0x1e regs 0x13/0x14 = 0x0404).

**Despite all of this, MAC's `BAR+0x04 bit 0` (link UP status) refuses
to latch.** Stays 0x00000b18 indefinitely.

Userspace experiments confirmed:
- BAR+0x04 is hardware-driven (write attempts trash the MAC into
  degraded state)
- 64+ random writes don't crash the kernel
- Force-100M no-AN via BMCR=0x2100 doesn't help
- Master reset / switch reset cycles don't help

## The question

**What does Orbis do at the MAC side (BAR0+0x000..0x3ff space) that
makes the MAC's link detector finally latch BAR+0x04 bit 0 = 1?**

Specifically dig:

### Q2 — Trace the link-latch path in Orbis

The Baikal MAC has BAR+0x04 bit 0 = "link UP" (per all prior agent
research).  Find the **inverse trace**:

- What CONDITION in MAC silicon must be true for bit 0 to set?
- Is there a MAC-side "link detect enable" bit?  Maybe an opcode
  somewhere that says "MAC, start monitoring PHY's link signal"
- Does Orbis write to BAR+0x080-0x0bf range during init?

### Q3 — PCS/sync registers

PHY-MAC interface PCS sync might be required.  Look for:
- BAR+0x118 / 0x128 / 0x12c (counters or status?)
- Any "PCS_SYNC" / "RX_SYNCED" / "MAC_READY" register naming
