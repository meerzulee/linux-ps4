### What's not working

The PHY itself reports **link DOWN at the physical layer**:

```
BMSR latch=0x7969 current=0x7969 (bit 2 = 0 in BOTH halves of double-read)
linkreg = 0x00000b18 (bit 0 = 0)
```

BMSR bit 2 is latched-low — first read returns the latched value,
second returns current state.  Both = 0 means link is currently down,
not just "was down at some point".

The PHY cycles:
```
AN incomplete (0x7949) → AN complete (0x7969) → AN incomplete (0x7949) → ...
```

This cycling is partly caused by our kthread restarting AN every 15s
when link is down.  But even when AN completes successfully, the
physical link signal never establishes.

## The question

**What does Orbis do POST-mts_mac_init that brings up the actual
physical link signal between MT7531 PHY and Baikal MAC?**

Specifically:
- We replayed `mts_mac_init` MAC-side BAR writes
- We replayed parent prelude
- We started RX engine
- We restart AN periodically
- PHY confirms AN-complete + partner advertising
- But BMSR bit 2 (link UP) NEVER sets

So Orbis must do something we're missing.  Candidates to investigate:

### Q1 — PMA / PCS enable register

The PHY's PMA (physical media attachment) and PCS (physical coding
sublayer) must be enabled for the PHY to drive the link.  Standard
Clause-45 MMD locations:
- MMD 1 (PMA/PMD): reg 0 = PMA_CONTROL_1 (bit 11 = power-down)
- MMD 3 (PCS): reg 0 = PCS_CONTROL_1

Does Orbis read or write to MMD 1 or MMD 3?  Decompile
`mts_mac_init` (FUN_c85ecb60) and `mts_ifup` (FUN_c85ec940) — look
for any C45 access to MMD 1 or 3.

### Q2 — Master/Slave configuration

We advertise 1000BT (reg 9 = 0x0200) but partner reg 5 = 0xc5e1
doesn't include 1000BT.  Reg 10 = 0x3c00 (or 0x7c00 earlier).  Bit 13
of reg 10 in 802.3 is "M/S config fault".

Does Orbis do anything to force master or slave mode in reg 9?
Bits 11-12 of reg 9 are M/S configuration.  Look for writes to reg 9
beyond OR-setting 0x0200.

### Q3 — RGMII / SGMII timing setup

The MAC↔PHY interface needs proper timing.  For RGMII this is in
vendor-specific MMD registers (e.g. mainline mtk-ge.c writes MMD 0x1e
reg 0x13/0x14 for TX delay).  For SGMII it's different again.

Does Orbis do any TX_DELAY / RX_DELAY / SKEW configuration?  Look at
all C45 writes in mts_mac_init that target MMDs other than 0x1e/0x1f
(those are Realtek-specific no-ops on MT7531).

### Q4 — Pseudo MII / interface mode select

The MT7531 has multiple PHY-MAC interface modes (MII, RMII, RGMII,
SGMII).  Selection might be in a vendor register.

Does Orbis write to anything that looks like an interface-mode select
register?

### Q5 — In-band switch management

glm-5.1's v83 dig found `FUN_c85f11c0` sending MDIO management frames
via the TX engine with ethertype `0xFA42` to switch ports 2/3.

Is this required for link to come up on the host port?  If yes, we
need to implement TX path before link.  If no (just for inter-switch
config), skip for now.

Look at `FUN_c85f2250` (gbe:ctrl init helper that calls FUN_c85f11c0).
What does it actually do to the switch?  Is the host port part of the
config?

### Q6 — Any other Orbis-side "force link up" mechanisms

Look for:
- Writes to BAR+0x04 that set bit 0 (we currently never write BAR+0x04)
- Functions named like `*_link_up*` or `*_carrier*`
- Any side-effect of bus_setup_intr / mts_intr first-call handling
