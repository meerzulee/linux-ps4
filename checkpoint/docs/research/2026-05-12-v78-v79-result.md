# v78/v79 hardware result — sky2 + Baikal SMI works, link still DOWN (2026-05-12)

Two iterations of the phase-2 sky2 strategy.  Both boot, bind cleanly,
create `enp0s20f1`, and do **not** emit `phy I/O error` (the v69 dead-end
symptom).  Neither one brings link UP.

## v78 — sky2 with custom SMI MDIO accessor at BAR+0x00

Patch: `patches/6.x-baikal/0700-network-sky2/0008-sky2-baikal-mts-smi-mdio.patch`

  - Routes `gm_phy_write` / `__gm_phy_read` through a Baikal-specific
    SMI helper at BAR+0x00 (cl22 protocol from Orbis RE).
  - Skips Yukon's `sky2_phy_init` for Baikal — replaces with BMCR
    soft-reset + AN restart only.
  - Restores Baikal entry in `sky2_id_table`.

Boot result (`checkpoint/uart-logs/2026-05-12_2016-v78-sky2-baikal-smi.log`
was actually the v78 capture):
```
[    4.403525] sky2 0000:00:14.1: Yukon-2 OptimaEEE chip revision 0
[    4.473043] sky2 0000:00:14.1 eth0: addr 2c:cc:44:57:0b:3a
[   23.145366] sky2 0000:00:14.1 enp0s20f1: renamed from eth0
[   25.540180] sky2 0000:00:14.1 enp0s20f1: enabling interface
```
**No `phy I/O error`**.  netdev came up admin-UP.  But `Link detected: no`,
ethtool reports `Speed: Unknown!`.  Live BAR readback after ~1 min uptime
shows the SMI controller stuck (every PHY register returns the same
echo-of-last-write data field — Orbis's mts_intr first-call enable for
`BAR+0x204` was missing).

## v79 — add Baikal IRQ block enable + trust Orbis PHY

Patch: `patches/6.x-baikal/0700-network-sky2/0009-sky2-baikal-irq-enable-skip-phy-init.patch`

  - Calls `sky2_baikal_irq_enable(hw)` from `sky2_probe` right after
    `sky2_reset(hw)`.  That writes `BAR+0x204 = 0x10001388` and
    `BAR+0x54 = 0x7bfffe` — mirrors Orbis's first-interrupt enable.
  - Drops the BMCR reset + AN restart we did in v78 — trust Orbis's
    pre-handoff PHY state (it had AN complete at handoff).

Boot result (`checkpoint/uart-logs/2026-05-12_2055-v79-sky2-baikal-irq.log`):
same dmesg shape as v78.  enp0s20f1 admin-UP, NO-CARRIER, link DOWN.

Live BAR readback at uptime ~1min:
```
BAR+0x04 linkreg     = 0x00000204
BAR+0x50 irq_status  = 0x00040002
BAR+0x54 irq_mask    = 0x007bfffe   ✅ (our v79 write)
BAR+0x204 enable     = 0x10001388   ✅ (our v79 write)
PHY reg N (any N)    = 0x2000       ❌ (SMI stuck — echoes write op bit)
```

`BAR+0x204` and `BAR+0x54` are STILL set — our writes persisted.
But the SMI controller is still stuck.  Writing `0x40002` back to
`BAR+0x50` (ack) clears bit 1, but bit 0x40000 is **sticky** — won't
clear even on write-1.  That bit is the "AN failed / MAC idle" event;
Orbis's mts_intr handles it by writing `BAR+0x204 = 0` (disabling) and
restoring the per-IRQ mask — i.e. the MAC transitions into a power-down
state intentionally.

## Why v79 still fails

Sky2's ISR (`sky2_intr`) reads the **Yukon-2 status registers at
offset 0x108** (B0_Y2_SP_ISRC family), not Baikal's BAR+0x50.  So when
the MAC raises bit 0x40000 (AN failed), MSI is delivered, sky2's ISR
fires, reads its own status reg (sees zero), exits without ack-ing
BAR+0x50.  The Baikal MAC stays in the "interrupt pending" state
indefinitely, the internal clock gating goes into a degraded mode, and
SMI loses its bus clock.

Confirmed by live observation:
- `BAR+0x50 = 0x00040002` — both bits 1 (some baseline status) AND
  0x40000 (AN failed) pending un-handled.
- ACK'ing them from userspace clears bit 1 momentarily but bit 0x40000
  reappears immediately.

## v80 plan — Baikal-specific IRQ pre-handler

Sky2's `sky2_intr` doesn't know about BAR+0x50.  The minimal v80 change:

```c
static irqreturn_t sky2_intr(int irq, void *dev_id)
{
    struct sky2_hw *hw = dev_id;

#ifdef CONFIG_X86_PS4
    if (sky2_is_baikal(hw)) {
        u32 stat = sky2_read32(hw, 0x50);
        if (stat)
            sky2_write32(hw, 0x50, stat);   /* ack all Baikal IRQ bits */
        /* fall through to sky2's normal handler in case there are
         * Yukon-2 bits to handle too.  Worst case the Yukon path
         * sees zeros at offset 0x108 and exits. */
    }
#endif

    /* ... existing sky2_intr body ... */
}
```

Open question: will ACKing bit 0x40000 actually unstick SMI?  We tested
this from userspace once — bit didn't clear.  But maybe the ack-clear
semantic only works from inside the ISR with the IRQ asserted at the
PIC level.  Worth testing in v80.

If v80 still doesn't fix SMI, deeper options:
1. RE the Orbis "gbe:ctrl" and "gbe:phy_ctrl" kthread functions to see
   what periodic work they do — maybe SMI requires a periodic
   refresh / heartbeat.
2. Replace sky2's IRQ handler entirely for Baikal with a Baikal-only
   handler that does NO Yukon-2 ack and tracks Baikal IRQ bits exactly
   like Orbis's mts_intr.
3. Disable IRQ-based delivery on Baikal and switch to pure polling for
   link state.
