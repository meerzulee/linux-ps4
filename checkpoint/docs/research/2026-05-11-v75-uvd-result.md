# v75 UVD VCPU bring-up — BO size ruled out; Ghidra re-RE exposes v74 mistakes

**Date:** 2026-05-11
**Patches:**
- `patches/6.x-baikal/0300-gpu-liverpool/0034-amdgpu-uvd-v4-2-liverpool-vcpu-start.patch` (v74 unchanged)
- `patches/6.x-baikal/0300-gpu-liverpool/0035-amdgpu-uvd-bo-size-liverpool-3-region.patch` (v75)

**Log:** `checkpoint/uart-logs/2026-05-11_1742-v75-uvd.log`

**Result:** ❌ Same VCPU timeout, **same VM fault page (0x300000 vmid 4)** as v74. BO size eliminated as a candidate. Subsequent Ghidra re-RE exposed real mistakes in v74's register sequence.

The "enlarging" gate fired (`2764 KB → 3168 KB`). The fault page (`3145728 = 0x300000`, byte address `0x300000000` = 12 GB) is unchanged. BO size is not the variable.

## What Ghidra revealed about v74's register sequence

Re-decompiled `uvd_vcpu_start_baikal` (FUN_c88f8610) and `uvd_kmd_hw_init_stage2` (FUN_c88f8cf0) and `uvd_ctx_dma_map_buffer` (FUN_c88eb570). Three concrete mistakes in v74:

### Mistake 1 — `LMI_VM_VBASE` should be 0, not `fw_gpu_addr`

Sony reads `uVar5 = ctx[+0x88]` then writes to LMI_VM_VBASE:
```c
gpu_reg_write(0x3c69, uVar5 & 0xffffffff);
gpu_reg_write(0x3c68, uVar5 >> 0x20);
```

**ctx[+0x88] is 0 at this point.** Proof:
- `uvd_kmd_hw_init_stage2` allocates 3 regions at ctx+0x50/+0x70/+0x90 — **never touches +0x88**.
- The only function that writes `ctx[+0x88]` is `uvd_ctx_set_field_88`, called as a callback from `uvd_ctx_dma_map_buffer`.
- `uvd_ctx_dma_map_buffer`'s only caller is `FUN_c88d7d50` in **`hdac.c`** (HDA audio), not the UVD path.

So Sony's `LMI_VM_VBASE = 0`. Our v74 writes `0xf00400000`. **This is the leading suspect for the VCPU hang.**

### Mistake 2 — `0x3da9` page-size encoding missing

Sony:
```c
uVar1 = log2_helper(0x1000);   /* = 12 */
uVar2 = log2_helper(0x100);    /* =  8 */
gpu_reg_write(0x3da9, (uVar1 & 0x1f) + 0x11010000 + (uVar2 & 0x1f) * 0x100);
/* Final value: 0x1101080c */
```

Our v74 writes a bare `0x11010000`. The `+ 0x080c` encodes 4 KB pages (log2=12) and 256 B sub-pages (log2=8). Without it, the DMA engine has zero idea of its page geometry.

### Mistake 3 — Spurious writes to `0x3d62` and `0x3d63`

v74 lines 207–211 write `0x44444444`-pattern to `0x3d62` and `0x3d63`:
```c
WREG32(0x3d62, (k << 28) | k | (k << 4) | ... | (k << 24));  /* = 0x44444444 with k=4 */
WREG32(0x3d63, (k << 24) | (k << 20) | ... | (k << 4));      /* = 0x04444440 */
```

**Sony's `uvd_vcpu_start_baikal` never touches 0x3d62 or 0x3d63.** These writes were misattributed from a different RE pass (possibly from a sibling Sony function like `uvd_vcpu_start_gladius`, which we have not yet compared register-by-register). They likely scribble garbage into VCPU configuration registers.

### Other notable Ghidra findings (non-mistakes, but useful)

- **Sony's UVD regions live at virtual 128–130 GB.** `uvd_alloc_region` (FUN_c88f94e0) calls `FUN_c86131c0(0, 2, 0, 0x2000000000, 0x2080000000, ...)`. So Sony's BOs are far away from our mainline-allocated 0xf00400000 (60.4 GB) AND from the fault page 0x300000000 (12 GB). The fault address remains an open question — it might be from a firmware-internal constant we're not yet seeing in the decompilation, or from stale state we haven't initialized.

- **mainline gmc_v7_0** *does* program vmid 4 PT base (line 681: shared with kernel root PT) and has a Liverpool/Gladius-specific block (lines 691–702) that sets `CONTEXT0_PAGE_TABLE_START_ADDR + i` to 0 and END to `max_pfn-1` for i in 2..7 (so vmid 4 has valid PT range). So vmid 4's *configuration* is sound; the *PT content* (page-entry for byte 0x300000000) is what's missing.

- **`*ctx[+0x4c]` is the chip-revision dispatcher**: 0=lvp_early, 1=Baikal, 2=Gladius. Set by `uvd_kmd_module_op` from `get_chip_family()`.

- **Sony firmware blob location** (rev=1 / Baikal): `&DAT_ffffffffc8c67ff0`, size `0x4ca38` (314 KB), banner `[ATI LIB=UVDFW,1.101.42]`. Matches the version we already extract via `wrap-uvd-firmware.py`.

## v76 plan

Three changes, smallest-step first:

### v76a — fix the three known mistakes, no architecture changes

```diff
- WREG32(0x3d62, ...);  /* spurious */
- WREG32(0x3d63, ...);  /* spurious */
+ /* removed: Sony's uvd_vcpu_start_baikal doesn't write 0x3d62 / 0x3d63 */

- WREG32(0x3da9, 0x11010000);
+ WREG32(0x3da9, 0x11010000 | 12 | (8 << 8));  /* = 0x1101080c, page-size encoding */

- WREG32(0x3c69, lower_32_bits(fw_gpu_addr));
- WREG32(0x3c68, upper_32_bits(fw_gpu_addr));
+ /* Sony writes ctx[+0x88] which is 0 (uvd_kmd_hw_init_stage2 never sets it). */
+ WREG32(0x3c69, 0);
+ WREG32(0x3c68, 0);
```

**Expected hardware signal in v76a:**
- 🟢 if VBASE=0 was the key: VCPU starts (STATUS bit 1 set), ring test attempt follows
- 🟡 if fault address moves: useful new data — we'd see VCPU accessing some other address that needs mapping
- 🔴 if fault unchanged at 0x300000000: VBASE isn't the variable, and we have to investigate firmware-internal addressing more

### v76b (if v76a still faults) — map the firmware BO into vmid 4 at the firmware's expected virtual address

If the VCPU is hard-wired to virtual 0x300000000 (the fault page), and Sony's GPUVM allocator placed the BO there in their setup, we need to do the same. This means programming GMC vmid 4's page-table-base register to point to a custom PT we maintain, with virtual `0x300000000..0x300308000` mapped to our firmware BO's physical VRAM pages.

This is a real amdgpu mod (custom vmid 4 page table), not in v76a's scope.

### v76c (followup) — fix the BO size correctly

v75's bo_size change is harmless and remains in. If/when v76b succeeds, we can revisit whether 3.5 MB is right (Sony allocates 3 separate regions; we conflate them in one BO).
