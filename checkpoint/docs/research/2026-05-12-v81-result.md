# v81 hardware result — kimi hypothesis was WRONG (2026-05-12)

## TL;DR

Skipping sky2's B0_IMSK writes on Baikal did NOT recover the SMI MDC.
It made things WORSE: in v80 we'd get at least one real PHY register
read latched in the SMI cmd register (`0x7949` = BMSR), in v81 we
never get any real read (data field is `0x2000` from the start).

So sky2's `B0_IMSK` write (= BAR+0x0c write to MAC_CTRL2/3 on Baikal)
was actually **briefly enabling MDC**, not killing it.  By gating it,
we removed the one thing that woke MDC up.

The real MDC killer is elsewhere in sky2's interface-up path — most
likely `sky2_mac_init`'s writes to GMAC offsets (port_base = 0x100,
+ GM_* offsets like 0x80) which on Baikal land in the BAR+0x180-0x200
range that we haven't reverse-engineered yet.

## v81 live readback after boot

```
BAR+0x04 linkreg     = 0x00000204     (same as v80)
BAR+0x50 irq_status  = 0x00000000
BAR+0x54 irq_mask    = 0x007bfffe     (Orbis state)
BAR+0x204 enable     = 0x10001388     (our v79 enable)
PHY reg N (any N)    = 0x2000         (stuck)
```

Arm-and-peek:
```
v80:  after ARM (no cmd): BAR+0 = 0x79490000   data = 0x7949  ← REAL BMSR
v81:  after ARM (no cmd): BAR+0 = 0x20000000   data = 0x2000  ← bogus
```

The latched `0x7949` in v80 was data from a successful sky2 link-timer
BMSR read that happened *after* sky2_open ran.  So MDC was clocking
even after sky2_open's B0_IMSK write.  The kill happens later, not
from B0_IMSK.

## Implication

The kimi/glm theory was right on the architectural picture (sky2's
Yukon-2 writes collide with Baikal's MAC control region) but wrong on
the specific killer.  B0_IMSK at BAR+0x0c isn't the trigger.

Likely candidates for v82:

1. **`sky2_mac_init` GMAC writes**: GM_TX_CTRL, GM_RX_CTRL,
   GM_SERIAL_MODE, GM_GP_CTRL, etc. all at `port_base + GM_*` =
   BAR+0x100 + 0x80..0x100.  Skip the entire function on Baikal.

2. **`sky2_hw_up` MAC-block writes**: status ring init writes that
   might touch BAR offsets in the 0xE0-0xF0 range (sky2's STATUS_*
   family).

3. **`sky2_phy_power_up`** even though we already skip
   `sky2_phy_init` — `sky2_phy_power_up` is a separate function
   that does PCI config writes via B2_TST_CTRL1.

The bigger structural question is what kimi flagged: **how much of
sky2's open path can we realistically negate before sky2-as-shell is
strictly worse than a fresh `ps4_mts.c` driver?**  v81 says: kimi's
estimate of "~80% negation needed" is starting to look real.

## Decision point for next session

Two paths:

A. **Continue sky2-shell**: v82 skips `sky2_mac_init` entirely on
   Baikal, plus possibly `sky2_hw_up`.  Risk: still doesn't fix MDC,
   and the netdev framework expects sky2's state machine to be
   running.

B. **Revert to standalone ps4_mts.c**: drop patches 0008–0011, restore
   v77/v77b ps4_mts driver, fix it properly using the full Orbis
   mts_mac_init + ISR sequence.  ~500 LOC clean driver vs ~50 LOC of
   sky2 negation that doesn't work.

Recommendation: pivot to (B) next session.  Sky2's downstream
init writes too many things at the wrong offsets to easily isolate.
The Orbis driver does proper MAC init in 50 register writes — we
should mimic it directly.
