A17 patch summary
-----------------

Hypothesis: STATUS bit 1 only fires AFTER the fw processes its first ring
packet. Mainline waits for bit 1 BEFORE submitting → deadlock.

A17 changes `uvd_v4_2_start_liverpool` so the if(!succeeded) branch
DRM_WARNs instead of returning -ETIMEDOUT, falling through to the
post-success ring-buffer setup. Lets amdgpu proceed to ring_test even
without STATUS bit 1.

Most likely root cause (best-guess synthesis)
---------------------------------------------

The fw's `mmUVD_STATUS` bit 1 doesn't fire because the **VCPU is waiting
on chip state set during Orbis runtime that we can't replicate from
Linux**. Candidates:

1. **SMU-set clock/voltage levels** for UVD's clock domain. Sony's
   gbase_update_vddnb (and similar functions) program SMU via SBL.
   Linux's amdgpu falls back to "no SMU" mode where chip stays at
   whatever level Orbis set before JB — likely too low/wrong for UVD.

2. **An SBL "you may use UVD" handshake** that Sony's userspace performs
   before the first decode. Sony's `sceSblDriverWriteSmuIx(0xc0200090, ...)`
   pattern (SAMU mailbox at GPU 0x22070-0x2207c) is the entry point.
   Without it, the security processor may keep UVD in a half-frozen
   state.

3. **Some bit/field in a register set ONCE at Orbis boot** that we never
   touch (gpu_reg_write(0xcc, 0); etc. in chip_specific_init are the
   only six writes; we don't replicate them).

These are all NOT REPLICABLE from mainline amdgpu running under
PSFree-Enhanced — they require SBL access, ATOM BIOS execution, or
chip-init code that lives in proprietary firmware blobs we don't have.

Conclusion: **UVD bring-up is structurally blocked**. Could potentially
be unblocked by:
- Porting Sony's SBL driver protocol to Linux (months of work)
- Reverse-engineering ATOM BIOS execution to set UVD-specific clocks
- Disassembling and patching the UVD microcode itself (would require
  Cadence Xtensa / Tensilica tooling)
