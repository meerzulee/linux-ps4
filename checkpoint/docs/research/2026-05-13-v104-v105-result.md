# v104 + v105 result — TX gate confirmed silicon-level (2026-05-13)

## TL;DR

Two final attempts (v104, v105) both falsified.  MAC's BAR+0x04 bit 0
latch is **not** triggered by:

| version | strategy | result |
|---|---|---|
| v104 | empty rings at engine STOPPED→RUNNING transition | `linkreg=0xb18` after 5s wait |
| v105 | re-run `mts_mac_init` in ndo_open with PHY already linked | `BAR+0x04=0xb18` immediately |

After v105, PHY-level state was fully healthy at the moment of mac_init:
BMSR=0x7969 (AN done), reg5=0xcde1 (partner ACK + 100/10), reg10=0x7800
(both receivers OK, LP_FD=1).  Multiple AN cycles fired (carrier_on/off
oscillation visible in dmesg).  **MAC absolutely refuses to latch
bit 0 of BAR+0x04 in any software-reachable state.**

## Final agent verdicts

- **Hermes** (`hermes-v105-mac-init-relocation.md`): Latch trigger is
  inside `mts_mac_init` itself.  Orbis calls it twice (attach + ifup).
  v105 implemented this — didn't latch.
- **Kimi** (`kimi-v105-latch-sample-window.md`): If `mts_mac_init` doesn't
  latch even with PHY pre-linked, the latch is **truly write-once per
  power cycle**.  v91 was a bootloader artifact.
- **GLM** (`glm-v105-init-move-patch.md`): Wrote the v105 patch
  implementation.  Risk LOW per glm.  v105 boot confirmed code applies
  cleanly — but the underlying latch did not fire.
- **Deepseek** (`deepseek-v105-final-bypass.md`): "Brutal answer: no
  software bypass exists.  Accept half-duplex.  Future kexec-snapshot
  from working Orbis to identify the missing register diff."

## What's been ruled out (cumulative across v82..v105)

- All BAR+0x034/0x038 kick values (bit 2 latch on 0x034 only)
- BAR+0x09c bit 6 toggle (destructive)
- BAR+0x200 master reset pulse (only fired RX_READY IRQs)
- BAR+0x138 register (write rejected, dead)
- BAR+0x0a00 TX arbiter (register doesn't exist)
- BAR+0x1c8 bit 6 (no effect)
- BAR+0x044 = tx_prod count
- BAR+0x080 various bits (kimi v99 TX_EN candidate)
- BAR+0x05c/0x060/0x064/0x06c/0x070 writes
- BAR+0x078 RX_GATE toggle
- BAR+0xe88/0xe8c/0xe84 (Yukon-2 prefetch registers — DON'T EXIST)
- MAC_ADDR0 program at 0x014/0x018
- Host link partner flap
- PS4 link flap (`ip link down/up`)
- AN restart from ndo_open (v103)
- Empty rings at engine restart (v104)
- mts_mac_init re-run in ndo_open (v105)

## Final state — half-duplex working driver

```
$ ssh ps4 ip -br link show enp0s20f1
enp0s20f1   UP   ee:57:74:42:33:75   <BROADCAST,MULTICAST,UP,LOWER_UP>

$ ssh ps4 cat /sys/class/net/enp0s20f1/carrier
1

$ ping -I enp8s0f3u1 -c 3 10.0.0.2  # host → PS4
$ ssh ps4 ip -s link show enp0s20f1 | grep RX:
    RX:  bytes packets errors dropped  missed   mcast
           180       3      0       0       0       0    ✅
```

- ✅ Netdev `enp0s20f1` registered
- ✅ Admin UP + LOWER_UP + carrier=1 (v94/v95 PHY-derived)
- ✅ NAPI RX delivery — host→PS4 packets reach netdev counters
- ✅ Link confirmed at 1Gbps Full Duplex from host RTL8153 side
- ❌ TX permanently blocked: BAR+0x06c bit 9 HW-locked while bit 0 of
  BAR+0x04 stays 0; TX engine refuses to fetch descriptors despite
  correct format (verified via v98 descriptor dump)

## Conclusion

After 14 patches + 5 major TX hypotheses falsified + 19 rounds of
agent research + ~180 live BAR-poke tests, the TX path is blocked at
a silicon-level interlock we cannot trigger from any post-power-on
software state.  v91's link UP was likely a bootloader-handoff
artifact that placed the MAC in a state we cannot recreate from
power-on without Orbis-side intervention.

The driver is a useful RX-monitor / packet-capture device.  Shipping
as half-duplex with documented TX limitation.
