## What actually unblocked it

The **3 critical pieces in combination**:

1. **Routing**: `IRQ_DOMAIN_FLAG_MSI_PARENT` + `msi_parent_ops` + AMDVI
   bus_token override (from v3-v7).  Routes child PCI MSI alloc through
   our domain.

2. **Standard composer**: kernel's `x86_vector_msi_compose_msg` writes
   real LAPIC encoding into device MSI cap (revert from v9's custom
   composer in v12).

3. **Southbridge programming**: `bpcie_config_msi` ported from 5.4
   Aeolia's apcie_config_msi, called from `bpcie_msi_write_msg` with the
   real LAPIC values.  Programs Baikal's per-function MSI redirection
   block at BAR2 + 0x110000 (v10 added the function, v11 fixed the
   func extraction).

4. **Bootarg**: `iommu=pt intremap=off` — IOMMU passthrough for DMA AND
   disable interrupt remapping.  intremap was blocking the southbridge's
   internal MSI delivery path (writes to 0xfdf8_xxxxxx range, which is
   Baikal's HT interrupt window — not the LAPIC range that intremap
   recognizes).

## On `bpcie_handle_edge_irq` = 0 (it's correct now)

The demuxer was needed for the 5.4 LEGACY MSI path where multiple
subfuncs shared a single LAPIC vector and the demuxer dispatched to
each.  In modern (6.x) per-device MSI with intremap-off, **each device
gets its own real LAPIC vector** programmed via `bpcie_config_msi`.
The kernel's standard `handle_edge_irq` runs and chains directly to the
driver's IRQ handler (xhci_irq, sdhci_irq, etc.) — no demux needed.

The `bpcie_handle_edge_irq=0` count we obsessed over for 11 iterations
was mostly a distraction.  The real test is **whether device drivers'
handlers fire**, which we now have direct positive evidence of: USB
enumerates, SATA works, amdgpu probes.

## Remaining issues (v13 candidates)

### A. amdgpu GPU command stream errors (6 events)

```
[118.012s] [drm:gfx_v7_0_priv_inst_irq] *ERROR* Illegal instruction in command stream
[118.731s] [drm:gfx_v7_0_priv_reg_irq] *ERROR* Illegal register access in command stream
[119.682s] [drm:cik_sdma_process_illegal_inst_irq] *ERROR* Illegal instruction in SDMA command stream
```

GPU init succeeds but command submissions fault.  Possible causes:
- amdgpu trying to use INTx (it didn't get MSI: "can't find IRQ for
  PCI INT A").  rmuxnet's c0066db41 commit forces MSI for Liverpool.
- PS4 GFX microcode/firmware not loaded
- IOMMU passthrough exposing GPU to addresses orbis didn't allow
- `[drm] Fence fallback timer expired` (3 events) — GPU work not
  completing in expected time

### B. ICC i2c timeouts (15 events)

```
[131.215s] baikal_pcie 0000:00:14.4: icc: interrupted or timeout: ret = 0
[131.222s] [drm:cq_exec] *ERROR* icc i2c commandqueue failed: -110
[131.228s] [drm:ps4_bridge_disable] *ERROR* Failed to disable bridge
```

bpcie's own ICC subfunc MSI not being delivered.  Ties to early
`call_irq_handler: 0.227 No irq handler for vector` message at t=5.86s.
ICC drives ps4_bridge for the GPU — this is BLOCKING GPU display init.

### C. amdgpu didn't get MSI

```
amdgpu 0000:00:01.0: can't find IRQ for PCI INT A; please try using pci=biosirq
```

amdgpu fell back to INTx.  rmuxnet's commit `c0066db41` ("amdgpu:
require MSI/MSI-X for PS4 Liverpool IRQs") forces MSI.
