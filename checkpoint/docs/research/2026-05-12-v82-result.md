# v82 hardware result — SMI MDC DEATH FIXED (2026-05-12)

## TL;DR

**The kthread-heartbeat hypothesis is CONFIRMED.**  v82 with parent
prelude + extended mts_mac_init + 3-second SMI BMSR kthread keeps the
SMI MDC clock alive **sustained**.  v78-v81 had MDC dead at ~1 minute;
v82 still has live SMI at 4+ minutes uptime.

Remaining problem is different and easier: physical link is DOWN despite
PHY's AN being marked complete.  No longer "the SMI dies" — now just
"PHY needs MT7531-specific AN/link bring-up."

## What changed in v82

1. `mts_parent_prelude()` — replays Orbis `FUN_c85131d0(softc, 1)`:
   - BAR+0xf10 = 1, then 2
   - BAR+0xf04 = 1, **12ms delay**, BAR+0xf04 = 2, **500ms delay**
     (switch chip GPIO reset with proper recovery time)
   - BAR+0x60 = 0x32100, BAR+0x64 = 6, BAR+0x68 = 0x63b9c, BAR+0x6c = 0x300
   - BAR+0x120 = 1, BAR+0x11c &= 0xf8ff
   - BAR+0x158 mode 2 → mode 1 (clock-domain latch)

2. Extended `mts_mac_init()` — full 25-write Orbis sequence vs v77's
   3-write minimal:
   - BAR+0x08 OR 0x07597c00 (MAC_CTRL1 enable bits)
   - BAR+0x0c clear bit 7 (MAC_CTRL2)
   - BAR+0x10 mask 0xffffff6e | 0x81 (MAC_CTRL3)
   - BAR+0x30 = 0x10100 (MAC_MODE)
   - BAR+0x74 = 0x2277 (MAC_PAUSE)
   - BAR+0x78 clear bit 0 (RX_GATE)
   - BAR+0x07c = 25MHz (MAC_CLK)
   - BAR+0x204 = 0x10001388 (full IRQ block enable)
   - BAR+0x54 = 0x7bfffe (per-IRQ mask)

3. `mts_phy_ctrl_fn()` kthread — reads BMSR via SMI C22 every 3s in a
   tight `while (!kthread_should_stop())` loop.  This is the SMI MDC
   heartbeat per glm-5.1 + kimi-k2.6 dig.

## Sustained SMI activity proof

At uptime 4 min, tight-poll of BAR+0x00 showed kthread transitions
every ~3 seconds:

```
t=0.0s   baseline  smi=0x79698100  (data=0x7969 from prior kthread read)
t=2.26s            smi=0x79694100  ← kthread wrote OP_C22_RD reg=1
t=2.26s            smi=0x79698100  ← SMI completed: data=0x7969 + DONE
t=2.84s            same pattern
t=4.31s            same pattern
```

6 transitions in 5 seconds = exactly the 3s cadence.  The `0x4100` is
`OP_C22_RD | (reg=1 << 8)` — BMSR read command.  The `0x8100` is the
post-completion state with DONE (bit 15) + bit 8 set.

Kthread visible as `[ps4_mts_phy_ctrl]` kernel thread at PID 76, state S
(sleeping in msleep_interruptible).

## Why is link still DOWN?

BMSR analysis at uptime 4 min:
- `0x7969` = 0b0111 1001 0110 1001
- bit 2 (link status) = **0** → link DOWN
- bit 5 (AN complete) = **1** → AN done
- bit 0/3/6/8/11/12/13/14 = ext cap, AN ability, MF preamble, ext status, 10BT/100BX modes

So PHY says AN done but link not present.  linkreg's speed field shows
`1000M full-duplex` (bits 2-3 = 10, bit 4 = 1) but bit 0 = 0.

Possibilities:
1. Cable not actually connected (try power-cycle the switch?)
2. MT7531 needs special vendor MMD writes for link to come up (mt7530-mdio.c
   in mainline does extensive MT7531-specific init)
3. Orbis `gbe:phy_ctrl` does AN restart on link-down — our heartbeat
   only READS BMSR, doesn't restart AN.  Need to extend kthread.

LP ABILITY (reg 5) = `0xc5e1` is interesting:
- 0xc5e1 = 0b1100 0101 1110 0001
- bit 0: 802.3 protocol selector
- bit 5: 10BT half
- bit 6: 10BT full
- bit 7: 100BX half
- bit 8: 100BX full
- bit 10: pause
- bit 14: ack
- bit 15: NP (next page)

So link partner DID advertise 10/100 + pause.  Cable IS connected,
partner IS speaking, but somehow link isn't latching.

## Next experiment (v83)

Extend the kthread to mirror Orbis `gbe:phy_ctrl` link-down handling:
- Every 3s: read BMSR
- If `(BMSR & 4) == 0` (link down): read reg 9 + 4, OR-set
  `reg 9 |= 0x0200` (advertise 1000-BT full) and `reg 4 |= 0x0180`
  (advertise 100BX), then write BMCR `|= 0x1200` (AN enable + restart).

This is the exact algorithm decompiled from `FUN_c85f0480` event 0x1
handler (glm-5.1 + kimi-k2.6 dig).

Pass criterion: linkreg bit 0 transitions to 1, dmesg shows
`link UP — 1000 Mbps full-duplex`.
