# v85 hardware result — bit 18 is unmaskable, PHY says link DOWN (2026-05-13)

## TL;DR

The histogram ISR + 5-second debug timer + BMSR double-read produced
*three* huge findings in a single boot:

1. **Bit 18 of BAR+0x54 (IRQ mask) is hardware-forced to 1.** We wrote
   `0x007bbffe` (bit 18 cleared) but readback shows `0x007bfffe`
   (bit 18 set) consistently across all telemetry dumps.  Either bit 18
   is read-only, write-1-to-set semantics, or the polarity differs from
   the other mask bits.

2. **The PHY says link is DOWN at the PHY level.**  BMSR double-read
   (`latch=0x7969 current=0x7969`) shows bit 2 = 0 in BOTH reads.  Bit 2
   is latched-low — first read returns latched-low value, second returns
   current state.  Both reads = 0 means **link is currently DOWN at the
   PHY**, not just latched.  The MAC isn't broken; the PHY isn't bringing
   up the link.

3. **Bit 18 IRQ pattern: `0x00040000` alone at ~4,600 Hz constantly**,
   with rare `0x00040040` (bit 18 + bit 6) and `0x00040060`
   (bit 18 + bit 6 + bit 5) events.  Bit 18 looks like a "MAC IRQ
   active" summary indicator that re-asserts level-style.  The real
   actionable events are the bit 5 and bit 6 patterns (5 and 1
   occurrences in 40 seconds).

## Implications

### Bit 18 is not the link blocker

Even if we COULD mask bit 18, the PHY would still say link is DOWN.
Masking the IRQ would just stop the IRQ flood; it wouldn't bring up
the link.  The 4,600 Hz IRQ rate is a symptom of the MAC engine running
with bad state, but it's not the root cause of "no link".

### The real blocker: PHY can't establish link

PHY reports AN complete (bit 5 = 1) but link down (bit 2 = 0).
Reading both BMSR bits in rapid succession (latch + current = both 0
for bit 2) confirms the PHY itself is the disconnect — not a latched
transient.

### The likely cause: MT7531 PHY PLL state was destroyed

From deepseek-v41's v83 research:

> mt7531_setup (DSA switch driver, NOT the PHY driver) enables the PHY
> core PLL:
> ```
> val = read_c45(control_phy, MDIO_MMD_VEND2, 0x403);
> val |= BIT(6) | BIT(4);   // RG_SYSPLL_DMY2 | PHY_PLL_BYPASS_MODE
> val &= ~BIT(5);            // clear PHY_PLL_OFF
> write_c45(control_phy, MDIO_MMD_VEND2, 0x403, val);
> ```
>
> Orbis does NOT write to MMD 0x1f register 0x403 in mts_mac_init.  The
> PLL might have been enabled by the Orbis msk_parent_init or by
> bootloader firmware.  If Linux reboots clean from Orbis (kexec),
> the PLL stays enabled.  But if we ever power-cycle or reset the
> switch (even accidentally via BAR+0xf04), the PLL state is lost
> and must be re-programmed.

Our `mts_parent_prelude` writes:

```c
writel(1, bar + MTS_PRELUDE_F04);
msleep(12);
writel(2, bar + MTS_PRELUDE_F04);
msleep(500);
```

This is the **switch chip GPIO reset** that — per deepseek — destroys
the MT7531 PHY PLL state.  After this reset, the PHY can still do AN
(the MDIO interface is alive) but it cannot generate a link signal
because its PMA/PCS clocking PLL is off.

This matches our observation EXACTLY:
- ✅ SMI MDIO works (PHY responds to register reads/writes)
- ✅ AN completes (BMSR bit 5)
- ✅ Partner advertises (reg 5)
- ❌ Link doesn't come up (BMSR bit 2 = 0, linkreg bit 0 = 0)

The MAC RX engine running etc. is irrelevant because the PHY can't
even deliver a link signal to the MAC.

## v86 plan

### Phase 1 (cheap test): skip parent prelude

If deepseek is right that parent prelude destroys PLL, simply NOT doing
the BAR+0xf10/0xf04/0x60..0x6c/0x158 prelude should leave the
bootloader's PLL state intact.  If link comes up after this, hypothesis
confirmed.

Risk: parent prelude might also do other necessary clock setup beyond
the PLL (since glm/gpt-5.5 found the BAR+0x158 mode 2→1 transition is
non-obvious).  We may need to do a PARTIAL prelude.

### Phase 2 (proper fix): implement C45 SMI accessor + PLL enable

Add `mts_smi_c45_read` and `mts_smi_c45_write` using the SMI protocol's
C45 opcodes (we have the defines: `MTS_SMI_OP_C45_ADDR=0x20`,
`MTS_SMI_OP_C45_RD=0xe0`, `MTS_SMI_OP_C45_WR=0x60`).

Then at end of `mts_mac_init`:

```c
u16 pll;
mts_smi_c45_read(mts, 0x1f, 0x403, &pll);
pll |= (1 << 4) | (1 << 6);  // RG_SYSPLL_DMY2 | PHY_PLL_BYPASS_MODE
pll &= ~(1 << 5);             // clear PHY_PLL_OFF
mts_smi_c45_write(mts, 0x1f, 0x403, pll);
```

Bonus: read MMD 0x1f reg 0x403 BEFORE and log the value, to confirm
the PLL is OFF (bit 5 set) when we enter mts_mac_init.

### Phase 3 (next agent dig if needed)

If even with PLL enable link still won't come up, the issue is
elsewhere.  Hand to fresh agent with this dump and ask: "what does
Orbis do POST-AN that brings up the actual MT7531 link signal?"
