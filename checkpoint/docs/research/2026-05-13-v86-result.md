# v86 hardware result — C45 protocol works but PLL was already ON (2026-05-13)

## TL;DR

**v86 SCORED TWO HUGE WINS:**

1. **C45 SMI accessor works correctly.**  First-ever successful C45
   transaction on PS4 Baikal MTS hardware.  Inferred Sony encoding
   (MMD device in bits 8-12, register address in bits 16-31) is
   confirmed by hardware response.

2. **PLL hypothesis (deepseek-v41) is falsified.**  MMD 0x1f reg 0x403
   read 0x1000 — bit 5 (PLL_OFF) was ALREADY 0.  PLL was on the whole
   time.  Our parent prelude does NOT destroy the MT7531 PHY PLL.

The real link blocker is elsewhere.  But we now have a working C45
accessor for future iterations.

## Boot evidence

UART log: `checkpoint/uart-logs/2026-05-13_0118-v86-c45-pll-enable.log`

### PHY PLL operation succeeded

```
[    5.010] ps4_mts 0000:00:14.1: PHY PLL: MMD 0x1f reg 0x403 read 0x1000 (bit 5 PLL_OFF=0)
[    5.018] ps4_mts 0000:00:14.1: PHY PLL: wrote MMD 0x1f reg 0x403 <- 0x1050 (PLL on, bypass mode)
[    5.035] ps4_mts 0000:00:14.1: PHY PLL: readback MMD 0x1f reg 0x403 = 0x1050
```

**Interpretation:** the C45 read succeeded, returned `0x1000` (just bit 12 set).
Bit 5 (PHY_PLL_OFF) was 0 — PLL was already on.  Our write OR-set bits 4
and 6 (RG_SYSPLL_DMY2 | PHY_PLL_BYPASS_MODE) and cleared bit 5.  Readback
confirmed `0x1050`.

But hardware state evolved identically to v85 after this — no link.

### Link state during boot

```
PHY ID = 0x03a29441 (MT7531 - identical to all previous boots)
reg 5 (LP ability) = 0xc5e1 (partner still advertising 10/100)
reg 9 (1000BT control) = 0x0200 (we advertise 1000BT)
reg 10 = 0x3c00 (was 0x7c00 in v85 - bit 14 cleared)
```

### BMSR oscillation pattern

```
[   99.290] BMSR latch=0x7949 current=0x7949 (link@phy=0 AN@phy=0)
[  102.384] BMSR latch=0x7969 current=0x7969 (link@phy=0 AN@phy=1)
[  114.516] BMSR latch=0x7949 current=0x7949 (link@phy=0 AN@phy=0)
[  117.592] BMSR latch=0x7969 current=0x7969 (link@phy=0 AN@phy=1)
[  129.786] BMSR latch=0x7949 current=0x7949 (link@phy=0 AN@phy=0)
[  132.896] BMSR latch=0x7969 current=0x7969 (link@phy=0 AN@phy=1)
```

**PHY cycles**: AN-complete → AN-incomplete → AN-complete → ...

Bit 2 (link status) stays 0 in BOTH halves of every double-read.
Link is NEVER physically established.  But AN keeps restarting and
re-completing.

This cycling is **caused by our kthread's AN restart every 15 seconds**
while link is down.  Each restart clears AN-complete bit, AN runs for
2s, completes, then kthread restarts AN again 13s later.

The PHY is doing its job (running AN, completing it, partner is
advertising) but the actual physical link signal never establishes.

## Hypotheses

### H1: Master/Slave config fault

We advertise 1000BT (reg 9 = 0x0200) but partner doesn't (reg 5 = 0xc5e1
only has 10/100 bits).  Normally this wouldn't trigger M/S resolution
(both sides agree 1000BT not available, fall back to 100/10).

But reg 10 in v86 = 0x3c00 = bits 10, 11, 12, 13 — bit 13 = **M/S
config fault** in IEEE 802.3.

Could be the PHY is stuck trying to do 1000BT master/slave even though
partner doesn't advertise.

**Fix to try:** stop advertising 1000BT (clear bit 9 in reg 9), so PHY
only does 100/10 negotiation.

### H2: We're disrupting AN by restarting too often

Our 15s AN restart cycle prevents AN from ever stabilizing into a
working link.  If we DON'T restart AN, the first AN completion might
actually bring up link.

**Fix to try:** remove the periodic AN restart entirely.  Only restart
AN on explicit link-down transition (currently we also restart every
5 iterations after initial down).

### H3: Force fixed 100M full duplex (skip AN)

If AN is fundamentally broken for whatever reason, forcing speed/duplex
manually bypasses AN.

**Fix to try:** BMCR = 0x2100 (bit 13 + bit 8 = 100M + full duplex,
AN disabled).

### H4: PLL bypass mode might be wrong for our case

Mainline `mt7531_setup` sets BYPASS_MODE (bit 6) when configuring as
**switch** — meaning the host provides external clock.  We're using
the PHY as a single-port (no switch), so the PLL should generate the
clock internally — bypass mode might be wrong.

**Fix to try:** don't set bit 6.  Write only bit 4, clear bit 5.

## v87 plan

Combine H1 + H2 (cheap, no risk):

1. **Stop advertising 1000BT** — write reg 9 with bit 9 cleared (= 0x0000)
   at end of mts_mac_init.  This forces PHY to only negotiate 10/100.
2. **Remove periodic AN restart** — kthread only restarts AN on the
   first iteration and on UP→DOWN transitions.  No more "retry every
   5 iterations" loop.
3. **Read reg 10 every kthread iteration** and log if value changes.
   This catches M/S config fault transitions.

If still doesn't work, try H3 (force 100M full) in v88.
