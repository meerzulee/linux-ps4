# v84/v84a hardware result — engines started, deepseek hypothesis FALSIFIED (2026-05-13)

## TL;DR

**v84a confirms:** the MAC RX engine being "running" (BAR+0x34 bit 0 sticky) is
NOT sufficient to make the MAC latch PHY link state to BAR+0x04 bit 0.
No link-change IRQ fires.  Link stays DOWN.

deepseek-v41's hypothesis from the 4-agent dig is **falsified**.  The C45
MMD path proposed by glm-5.1/gpt-5.5/kimi-k2.6 is now the prime remaining
suspect.

## v84 → v84a iterations

**v84 (md5 782c1563...)** failed at probe with `request_irq(0) failed: -16`.
Root cause: on PS4 Baikal, `pdev->irq = 0` because bpcie manages MSI via a
custom IRQ domain, not via standard PCI MSI initialization.

**v84a (md5 0147662f...)** fix: call `pci_alloc_irq_vectors(pdev, 1, 1,
PCI_IRQ_MSI | PCI_IRQ_INTX)` + `pci_irq_vector(pdev, 0)` before
`request_irq`.  bpcie's `bpcie_msi_write_msg` runs for our device after
this, and `pdev->msi_enabled` becomes true.

## Why deepseek's hypothesis was wrong

deepseek-v41's analysis said: "Without the RX engine running, the MAC does
not monitor for link state changes — BAR+0x04 bit 0 (link up) never gets
set."

We started the RX engine (bit 0 sticky in BAR+0x34).  Link still didn't
latch.  So either:
1. The engine isn't really "running" in the way the MAC needs (perhaps
   needs valid descriptors with specific format, not just non-null DMA
   addresses)
2. The MAC's link monitor is independent of the RX engine
3. There's another piece of init missing entirely

Looking at deepseek's exact claim: "the MAC does not monitor for link
state changes" was speculative — they couldn't have tested it.  The
empirical evidence now says engine bit 0 = 1 doesn't fire link-change
IRQs.

## Remaining unexplored leads

1. **C45 MMD writes** (glm + gpt-5.5): the "unconditional tail" of
   mts_mac_init that we never replicated:
   - `mmd 0x1e reg 0x189 = 0x110`
   - `mmd 0x1e reg 0x122 = 0xffff`
   - `mmd 0x1f reg 0x268 = 0x07f4`
   - `mmd 0x07 reg 0x3c = 0`  ← could be MT7531-related
   - `mmd 0x1e reg 0x33 clear bit 0x1000`
   - kimi argues these are "no-ops on MT7531" — but we haven't tested it

2. **BAR+0x04 R/M/W mask** (glm step 34 + 47): clear bits 8-9 + 12-13
   before/after PHY init. We never write BAR+0x04 in our driver.

3. **In-band switch management** (glm finding via `FUN_c85f11c0`): the
   MT7531 needs MDIO-via-ethernet-frame management with ethertype 0xFA42
   to configure switch port routing. This requires the TX path to be
   FUNCTIONAL (not just engines started with empty rings).

4. **PHY link-status is latched-low** (BMSR bit 2): a single BMSR read
   returns the latched value, not current state.  We've only confirmed
   `bit 2 = 0` from a SINGLE read.  Could mean link was down at some
   point since last read — not necessarily "currently down".  Need
   read-twice diagnostic.

5. **Realtek-style PHY pokes** (glm + kimi): the page 0x52b5 token-ring
   pokes.  Both glm and gpt5.5 documented them; kimi argues they're
   no-ops on MT7531.  Untested.

## v85 plan

**Phase 1: cheap diagnostics (2-line patches)**
- Add a second BMSR read after the first in `mts_phy_an_restart` and log both. If they differ, link is actually transient and AN is partially working.
- Read and log BAR+0x04 inside `mts_phy_ctrl_fn` every iteration to see if it ever transitions during AN restart.

**Phase 2: implement C45 SMI accessor**
- Add `mts_smi_c45_read` and `mts_smi_c45_write`.
- Use the SMI protocol's C45_ADDR/C45_RD/C45_WR opcodes (0x20/0xe0/0x60).

**Phase 3: add the unconditional MMD writes**
- Replicate gpt-5.5's tail at end of `mts_mac_init`.
- Test if any of these affect link bring-up.

**Phase 4 (if 1-3 don't work): MT7531 vendor PLL enable**
- `MMD 0x1f reg 0x403`: set bits 4, 6, clear bit 5
- This is the only MT7531-specific write deepseek flagged as NOT done by Orbis (might be set by bootloader, lost if we ever reset BAR+0xf04 properly)
