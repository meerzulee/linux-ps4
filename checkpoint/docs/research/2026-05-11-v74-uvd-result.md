# v74 UVD VCPU bring-up — diagnostic results, root cause isolated

**Date:** 2026-05-11
**Patch:** `patches/6.x-baikal/0300-gpu-liverpool/0034-amdgpu-uvd-v4-2-liverpool-vcpu-start.patch`
**Log:** `checkpoint/uart-logs/2026-05-11_1652-v74-uvd.log`
**Result:** ❌ VCPU still does not start, but diagnostics confirm **the register writes themselves are correct** — the bug is one level higher (the UVD BO is too small).

## What v74 added

Two things on top of v73:

1. **Call mainline `uvd_v4_2_mc_resume(adev)`** before Sony's cache-region writes. Hypothesis: mc_resume programs `mmUVD_LMI_ADDR_EXT` and the cache offset/size registers; without it, the VCPU's address-extension is left in whatever state the GPU reset left it. We then deliberately overwrite mc_resume's cache offset/size with Sony's values (Sony's are tuned for Liverpool's 3-region 3.1 MB layout).
2. **Diagnostic DRM_INFO at entry + DRM_ERROR on timeout**, dumping:
   - `fw_gpu_addr` (the GPU address we computed and wrote)
   - `LMI_VM_VBASE_HI/LO` readback (what actually landed in registers)
   - `LMI_EXT40_ADDR` readback (mc_resume's contribution)
   - `STATUS`, `SOFT_RESET`, `RBC_RB_CNTL` at the moment of timeout

## Observed signals (from boot log)

```
[9.985] [drm] ps4 uvd: liverpool start: fw_gpu_addr=0xf00400000
                vcpu_bo=000000006c49920d ring_gpu_addr=0x62f000
[12.013] [drm:uvd_v4_2_start.cold] *ERROR* ps4 uvd: Liverpool VCPU did not start
                (no STATUS bit 1 after 2s)
[12.034] [drm:uvd_v4_2_start.cold] *ERROR* LMI_VM_VBASE_HI=0x0000000f
                LMI_VM_VBASE_LO=0x00400000 (expected fw_gpu_addr=0xf00400000)
[12.046] [drm:uvd_v4_2_start.cold] *ERROR* LMI_EXT40_ADDR=0x8009000f
[~12.05] VM fault (0x02, vmid 4, pasid 0) at page 3145728, read from 'UMC'
```

### Decoding

| Quantity | Value | Interpretation |
|---|---|---|
| `fw_gpu_addr` | `0xf00400000` | High-VRAM region, 64.3 GB into GPU address space. Sensible — this is where `amdgpu_uvd_sw_init` placed the UVD BO. |
| `LMI_VM_VBASE_HI` | `0x0000000f` | Upper 32 bits of `0xf00400000`. **Exactly what we wrote.** |
| `LMI_VM_VBASE_LO` | `0x00400000` | Lower 32 bits of `0xf00400000`. **Exactly what we wrote.** |
| `LMI_EXT40_ADDR` | `0x8009000f` | Set by `uvd_v4_2_mc_resume()`. Bit 31 enable, high nibble `0xf` matches `fw_gpu_addr >> 32`. |
| VM fault page | `3145728` = `0x300000` | **3 MB exactly.** VMID 4 = UVD's MMU context. |

### What the data proves

1. **Our register writes are not the problem.** The `WREG32(0x3c68/0x3c69)` pair landed exactly where we computed (`fw_gpu_addr` echoed back via readback).
2. **mc_resume cooperated.** `LMI_EXT40_ADDR` is populated correctly with the matching upper nibble.
3. **The VCPU IS executing.** A VM fault from `vmid 4` means the firmware loaded, the VCPU is running, and it's issuing memory accesses through its MMU context.
4. **It faults trying to read past our BO.** Page `0x300000` = 3 MB. Our UVD BO is much smaller (see math below).

## Why this happens — the actual root cause

`amdgpu_uvd.c:323` allocates the UVD BO with:

```c
bo_size = AMDGPU_UVD_STACK_SIZE     /* 200 KB */
        + AMDGPU_UVD_HEAP_SIZE       /* 256 KB */
        + AMDGPU_UVD_SESSION_SIZE * adev->uvd.max_handles  /* 50 KB * 40 = 2000 KB */
        + AMDGPU_GPU_PAGE_ALIGN(fw_size + 8);              /* ~315 KB for Sony's blob */
```

Total: **~2771 KB ≈ 2.7 MB**. The BO occupies `[0xf00400000, 0xf006B3000)` in vmid 4.

Sony's Orbis kernel (RE'd from FUN_c88f8610) allocates **three** GPU buffers totalling **3.1 MB** for UVD on Liverpool:

| Sony region | Size | ctx field | Notes |
|---|---|---|---|
| Region 1 (firmware/code) | 1920 KB (`0x1E0000`) | `ctx[+0x80]` → register `0x3d83` size, ... | Holds the ucode blob proper. |
| Region 2 (workspace/stack) | 1168 KB (`0x124000`) | `ctx[+0x88]` → `0x3c68/0x3c69` LMI VBASE | What our v73/v74 writes target. |
| Region 3 (message ring/handles) | 16 KB (`0x4000`) | `ctx[+0x90]` → register ??? | Per-session handle table. |

When Sony's firmware initialises itself, it walks all three regions — and the VCPU's MMU translates those accesses through `LMI_VM_VBASE`. **It expects the contiguous 3.1 MB of regions 1+2+3 to all be mapped, starting at VBASE.**

Our single 2.7 MB BO is **~400 KB short** of covering Sony's region-3 territory. The fault at offset `0x300000` (3 MB from VBASE) lands right at the boundary where region 3 should start — and there's no GMC page-table entry there → VM fault.

## Why v74's `mc_resume` call didn't help

`uvd_v4_2_mc_resume()` writes the cache *offset/size* registers for the standard Bonaire-class layout, which is much smaller than Sony's 3-region layout. It also writes `mmUVD_LMI_ADDR_EXT` (which the diag print confirms is correctly set). Our Sony-overrides at lines 280-310 of the Liverpool function then bash the cache offset/size back to Sony's values (`0xfa00`, `0x40000`, `0x17a00`, `0x120800`) — which is exactly what we want.

So mc_resume's net contribution is the address-extension register, which IS correct. The hypothesis "v73's missing 0x3c68/0x3c69 plus missing mc_resume" was right about the registers but wrong about why VCPU still fails. The deeper issue (BO too small) wasn't visible until diagnostics came in.

## v75 plan — enlarge the UVD BO for Liverpool

Smallest patch that should fix this:

```c
/* amdgpu_uvd_sw_init, near line 323 */
if (adev->asic_type == CHIP_LIVERPOOL || adev->asic_type == CHIP_GLADIUS) {
    /* Sony's UVD layout: region1 (1920K) + region2 (1168K) + region3 (16K),
     * all referenced by the firmware via LMI_VM_VBASE. Must be ONE
     * contiguous BO so the VCPU's MMU translation works across regions. */
    bo_size = ALIGN(0x1E0000 + 0x124000 + 0x4000 + 0x10000, PAGE_SIZE);
    /* 0x308000 + 0x10000 slack = 3.5 MB, safely above 3.1 MB */
} else {
    bo_size = AMDGPU_UVD_STACK_SIZE + AMDGPU_UVD_HEAP_SIZE
            + AMDGPU_UVD_SESSION_SIZE * adev->uvd.max_handles;
    if (!adev->uvd.address_64_bit) {
        bo_size += AMDGPU_GPU_PAGE_ALIGN(le32_to_cpu(hdr->ucode_size_bytes) + 8);
    }
}
```

**Expected v75 signal:** if BO size is the real blocker, the fault at page `0x300000` disappears. We then see either (a) VCPU sets STATUS bit 1 — success — or (b) a new fault at a *higher* page, meaning there's a fourth region we haven't accounted for yet (less likely; Sony's RE shows exactly 3).

If success, ring-test should follow — and that's its own separate question (the ring write/read pointer registers we have from v72/v73 are correct, but only matter once VCPU is alive).

## Followup if v75 still fails

- **(B) Check ctx[+0x90] register dispatch.** Sony writes region 3's GPU address to *some* register — we haven't identified which one yet. If the VCPU needs region 3 mapped at a *different* offset than (region1 + region2), just enlarging the BO won't help; we'd need to also write the region-3 base. This is what `mcp__ghidra` "find caller of `uvd_ctx_dma_map_buffer`" was going to dig into pre-context-compaction.
- **(C) Match Sony's `0x3d3c` / `0x3d62` / `0x3d63` writes more carefully.** These three set the GPUVM page-table base for vmid 4 directly. v73/v74 do write them, but the value computed from `((adev->vm_manager.vram_base_offset + addr) >> 12)` may differ from what Sony writes. Would require comparing readback against expected value.

## What v74 cost / saved

- Confirmed: register sequence is correct.
- Eliminated: "wrong value being written to LMI VBASE" as a candidate explanation (was the leading suspicion after v73).
- Identified: BO size is too small for Sony's 3-region layout.
- Cost: one PS4 reboot, ~5 min build, the diagnostic-print scaffolding (which we keep — useful for v75/v76).
