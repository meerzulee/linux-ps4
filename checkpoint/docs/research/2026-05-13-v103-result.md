# v103 boot result — Orbis-style AN restart from ndo_open (2026-05-13)

## TL;DR

v103 inserts the Orbis `gbe:phy_ctrl` event-1 AN-restart sequence into
`ndo_open` (after engine reset + RX ring prefill).  Goal: trigger a
fresh PHY/MAC link transition that the MAC's one-shot bit-0 latch can
catch.

**Boot result: PHY recovers cleanly, MAC bit 0 STILL refuses to latch.**

## Boot signatures observed

```
ps4_mts: v103: ndo_open AN restart BMCR 0x1000 -> 0x1200 (provoke fresh link-change for MAC bit 0 latch)
ps4_mts: v94/v95: netif_carrier_off (BMSR=0x7949 LPA=0xcde1)     ← PHY dropped link
ps4_mts: phy_ctrl: BMSR latch=0x7969 current=0x7969 (link@phy=0 AN@phy=1)
ps4_mts: phy_ctrl: 1000BT_STAT (reg10)=0x7800 ... L_RX_OK=1 R_RX_OK=1 LP_FD=1
ps4_mts: v94/v95: netif_carrier_on (PHY-level: BMSR=0x7969 LPA=0xcde1)  ← PHY came back
```

PHY did fresh AN, partner ACKed, both receivers OK.  netif_carrier went
off → on (v94 kthread tracked the transition).  **But:**

- `BAR+0x004 = 0x00000b18`  (bit 0 NOT set — MAC link bit unlatched)
- `BAR+0x06c = 0x00000100`  (bit 9 NOT set — TX DMA ready gate closed)
- `BAR+0x03c = 0x0103e000`  (TX ring base, never advances)
- IRQ histogram still only `0x00040000` (bit 18 noise) — **no bit 2 LINK_CHANGE**

## RX still works perfectly

180 bytes / 3 packets received in v103 boot.  Same NAPI delivery path
as v97.  TX dropped = 5 (from ARP attempts that never went on the wire).

## What this proves

The MAC's bit-0 latch is not provoked by:
- Driver-side AN restart via SMI (v103, this boot)
- Host link partner flap (v97 live test)
- PS4 ip link down/up (v97 live test)
- BAR+0x200 master reset pulse (v97 live test)
- BAR+0x1c8 bit 6 write per hermes hypothesis (v97 live test)
- BAR+0x09c bit 6 toggle (destructive — only TX-error recovery per
  hermes v99)
- BAR+0x044 writes (any value)
- BAR+0x080 various bits (kimi v99 TX_EN hypothesis)
- MAC_ADDR0 program (0x014/0x018 = netdev MAC)
- 0x05c/0x064/0x06c writes
- BAR+0xe88/0xe8c/0xe84 (Yukon-2 prefetch registers — confirmed
  DON'T EXIST on Baikal, all writes ignored)

The latch absolutely refuses to fire in any post-v97 state.

## Convergent agent diagnosis (v99-v102 rounds)

Three independent agents (hermes/kimi/deepseek) all converge:

1. `BAR+0x06c` bit 9 = TX-DMA-ready HW status (NOT driver writable)
2. Bit 9 gates open when MAC's BAR+0x04 bit 0 latches to 1
3. MAC bit 0 latch is one-shot edge-triggered on internal PCS/RGMII
   "link up" transition
4. In v91 boot the latch fired (caught PHY transition during empty-RX
   probe phase, BEFORE ndo_open existed)
5. In v97+ boots, the latch window expires before our ndo_open finishes
   priming RX; MAC commits to "no link" permanently

## Three options forward

### Option A: Move all init to ndo_open (kimi-style)

Per kimi v102, Orbis runs `mts_mac_init` inside `mts_ifup` (open path),
not probe.  Could replicate: move `mts_parent_prelude` + `mts_mac_init`
from probe to ndo_open.  Latch window then opens when interface is
brought up, RX prefill happens after but the MAC stays "monitoring".

Pros: matches Orbis structure exactly.  Risk: substantial reorder, may
break PHY init or first-boot link altogether.

### Option B: Accept half-duplex driver, commit current state

RX is fully functional.  Half a driver is useful (capture inbound
traffic, monitor mode).  Commit as a known-limitation milestone, ship,
move on to other PS4 Linux work.

### Option C: Different probe-time RX state

Keep all init in probe (working), but in probe leave RX ring with
*invalid* descriptors (matching v91 empty state) until ndo_open.  Force
the same "RX-busy free" window that v91 had.  Test if MAC latches.
