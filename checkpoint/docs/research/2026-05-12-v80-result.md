# v80 hardware result — ISR ack works, SMI still stuck (2026-05-12)

## Result summary

v80 added a pre-handler in `sky2_intr` that reads `BAR+0x50` (Baikal
IRQ status) and W1Cs any pending bits before sky2's Yukon-2 path runs.
Returns IRQ_HANDLED instead of IRQ_NONE on Baikal when only Baikal bits
were pending.

UART log: `checkpoint/uart-logs/2026-05-12_*-v80-sky2-baikal-isr-ack.log`

Hardware result:
  ✅ ISR ack works.  BAR+0x50 stays at 0x00000000 in live readback
     (v79 saw bit 0x40000 sticky-pending).
  ✅ sky2 binds + creates enp0s20f1, no phy I/O errors
  ✅ All Orbis MAC init values preserved across handoff:
     BAR+0x08 = 0x0f597c00, BAR+0x30 = 0x10100, BAR+0x74 = 0x2277,
     BAR+0xac = 9, BAR+0x200 = 0, BAR+0x204 = 0x10001388,
     BAR+0x54 = 0x7bfffe, BAR+0x07c = 25M (clock ref)
  ❌ SMI controller still stuck — all PHY register reads return raw
     `0x20008000` (DONE bit set + WR_OP residue 0x2000 in data field)
  ❌ Link DOWN, NO-CARRIER, ping 192.168.50.95 fails

## Root cause refined

The IRQ ack was a real fix for the pending-IRQ side effect, but it
did NOT fix the SMI controller.  SMI is stuck "responding" to reads
with the residue of the prior write operation (echoing the WR_OP bit
0x2000 instead of fetching real PHY register data).  The MAC's MDIO
bus clock isn't running.

The MAC's full init register state from Orbis IS preserved — none of
the Orbis-init bits got cleared by sky2.  So the regression isn't
from sky2 overwriting Orbis's init values.

What sky2 might be doing that breaks SMI clock:
  1. **`sky2_reset(hw)` runs Yukon-2 reset sequence** — writes B0_CTST
     (sky2's offset 0x4) which on Baikal is some MAC control register
     (0x4 = link_status reg per our RE — we saw 0x204 = idle).  Other
     B0_* writes at offsets 0x8, 0xC, 0x10 hit Baikal's MAC_CTRL1/2/3.
     Specific bits sky2 toggles might cut SMI clock gating even while
     leaving the bytewise values intact.
  2. **Sky2's `sky2_init_hw` writes a long sequence** — e.g. B2_MAC_*
     at offset 0x100+ which on Baikal is the secondary MAC addr region;
     also B2_TX_*/B2_RX_* at higher offsets.
  3. **Sky2's link timer fires periodic PHY reads** — possibly each
     SMI read leaves the controller in a state where the next read
     finds garbage; over time, the cycle locks the bus.

## What to try next (v81 candidates)

A. **Skip sky2_reset on Baikal**.  Add an `if (sky2_is_baikal(hw))
   return;` at the top of `sky2_reset()`.  Risky because sky2_open
   assumes sky2_reset has run, but worth testing — if SMI stays alive
   after probe, we've identified the culprit class.

B. **Compare BAR state pre- vs post-sky2_reset**.  Add `dev_info`
   prints right before and right after `sky2_reset()` that dump
   BAR+0x08/0xC/0x10/0x30/0x9c/0x204.  See exactly what sky2_reset
   changes.

C. **Restore Orbis state after sky2_reset**.  At end of `sky2_baikal_irq_enable`,
   also re-write the full Orbis MAC init sequence (mts_mac_init regs).
   Costly but tests whether re-instating Orbis state recovers SMI.

D. **Disable sky2's link timer on Baikal** — `del_timer_sync(&hw->watchdog_timer)`.
   If SMI degrades only over time (link timer reads), this stops it.

Best first test is (B): observe what sky2_reset actually changes.

## Open question

The PHY reads at sky2_probe time DON'T log "phy I/O error" — meaning
the SMI WAS working at probe.  Sky2 successfully read PHY registers,
saw whatever it expected (or didn't validate strictly).  The break
happens between probe success and our live mmap probe ~5 min later.

A simple instrumentation patch would tell us: instrument
`sky2_baikal_smi_read` to log the raw `v` value every Nth call, and
see at what time the data field flips to constant 0x2000.
