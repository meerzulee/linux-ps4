# v87+ — what-if hypothesis grid (post-v86)

Generated 2026-05-13 to drive ~10 iterations of what-if exploration
while v87 research agents work in parallel.  Each row is a hypothesis
+ minimal test patch + expected signal.

## Confirmed wins (don't re-test)

- SMI MDIO C22 + C45 work
- PLL is on (MMD 0x1f reg 0x403 bit 5 = 0)
- AN completes (BMSR bit 5 cycles 0→1)
- Partner advertising (reg 5 = 0xc5e1)
- Parent prelude + mts_mac_init runs
- ISR + MSI vector + engines started

## Confirmed dead ends

- BAR+0x54 bit 18 unmaskable (hardware forces it back to 1)
- Engine bit 0 alone doesn't latch link
- PLL was the wrong hypothesis (was on)
- Realtek MMD writes are no-ops on MT7531

## Live hypotheses (test next, ordered by likelihood)

### H1: Stop our 15s AN restart cycle
**Why:** v86 showed BMSR oscillating between 0x7949 ↔ 0x7969 — we're
forcing AN to restart every 15s before link can stabilize.

**Patch:** kthread only restarts AN on first iteration + UP→DOWN
transitions.  Drop the "every 5 iterations while down" branch.

**Test:** dmesg shows fewer AN restart events; BMSR settles into a
stable value (either 0x7969 with link UP or some other state).

### H2: Stop advertising 1000BT
**Why:** Partner reg 5 = 0xc5e1 only advertises 10/100.  Reg 10 = 0x3c00
had bit 13 set in v85 = M/S config fault.  We advertise 1000BT (reg 9
= 0x0200) which forces M/S resolution that can't succeed with this
partner.

**Patch:** Write reg 9 = 0x0000 (no 1000BT) at end of mts_mac_init,
BEFORE the BMCR AN restart.

**Test:** Reg 10 no longer has bit 13 set; AN converges to 100BX
full duplex; linkreg bit 0 sets.

### H3: Force 100M full duplex without AN
**Why:** If AN itself is broken in some way we don't understand,
bypass it.  Force speed/duplex.

**Patch:** Write reg 9 = 0, reg 4 = 0 (no advertise), BMCR = 0x2100
(AN disabled, speed 100M, full duplex).  Skip our AN-restart kthread
path.

**Test:** dmesg shows BMCR latched 0x2100 (no 0x1200 restart pattern);
linkreg should show speed=01 (100M) bit 4 set (full), bit 0 set
(link up).

### H4: BAR+0x04 needs explicit clear bits
**Why:** glm's v83 dig found Orbis does R/M/W on BAR+0x04 twice in
mts_mac_init: `val & 0x7FFFCFFF` (clear bits 8-9 + 12-13 + 31).  We
never touch BAR+0x04.

**Patch:** Add `writel(readl(bar+0x04) & 0x7FFFCFFF, bar+0x04)` at end
of mts_mac_init.

**Test:** linkreg readback shows speed/duplex bits cleared, then
re-fills when link comes up.

### H5: Skip parent prelude entirely
**Why:** If we destroy something via BAR+0xf04 (switch GPIO reset) or
BAR+0x158 mode 2→1 that we don't understand, skipping the prelude
might rely on bootloader state.

**Patch:** Comment out `mts_parent_prelude(mts)` call in mts_probe.

**Test:** Does link come up?  Side effect: parent reset might be
needed for other things, so SMI might break.  Reversible if needed.

### H6: BAR+0x54 mask polarity inverted
**Why:** Hardware forces bit 18 = 1.  Maybe polarity is "1 = mask out"
(disable), so 0x7bfffe means "mask out everything except bit 0, 18,
22, 27-31".  Bit 18 firing wouldn't make sense then.  But what if
the IRQ source for bit 18 ignores the mask entirely?

**Patch:** Try BAR+0x54 = 0x00040000 (only bit 18 unmasked) and see
if other IRQs disappear (confirming polarity).  Or try 0xffffffff
(all bits) and see if bit 2 link-change starts firing.

**Test:** Histogram of irq_status patterns changes; specifically
look for bit 2 (0x4) appearing.

### H7: TX engine kick is fire-and-forget — needs periodic re-kick
**Why:** v84a saw BAR+0x38 momentarily = 0x8 then = 0x0.  Maybe TX
engine needs periodic kicks while waiting for descriptors.

**Patch:** Kick BAR+0x38 |= 1 every kthread iteration.

**Test:** Histogram shows new IRQ patterns; or link comes up.

### H8: MT7531 needs vendor-specific RGMII timing
**Why:** mainline mt7530-mdio.c does extensive RGMII delay
configuration via MMD vendor regs.  Without those, the MAC↔PHY
data lines have wrong skew, no link.

**Patch:** Pick a known mainline RGMII config and replay via C45.
Specifically MMD 0x1e reg 0x13 and 0x14 with delay values.

**Test:** Link should establish.

### H9: PMA / PCS power-down (MMD 1 or 3 reg 0 bit 11)
**Why:** Standard Clause-45 MMD 1 (PMA) reg 0 bit 11 is power-down.
If set, PHY won't transmit.  We've never read it.

**Patch:** C45 read MMD 1 reg 0; if bit 11 set, clear it.  Same for
MMD 3.

**Test:** dmesg shows MMD 1 reg 0 value; link should establish if
this was the issue.

### H10: BAR+0x78 RX_GATE bit 0 was reverse-meaning
**Why:** mts_mac_init clears bit 0 of BAR+0x78 ("enable RX gate" per
RE doc).  But "RX gate" might mean "gate (block) RX traffic" — i.e.,
clearing = open, setting = close.  We're matching Orbis but maybe
they intentionally block during init and unblock later.

**Patch:** Set bit 0 of BAR+0x78 after engine start.

**Test:** Maybe link state changes.

## Strategy

After agents return v87 findings, prioritize:

1. **If agents identify a specific MMD/register write we missed:** that
   becomes v87. Other what-ifs become v88+.
2. **If agents don't find a clear answer:** start with H1+H2 in v87
   (cheapest, lowest-risk, no Ghidra needed). Then H3 in v88. Then
   the more invasive ones.
3. **Use each iteration to test 1-2 hypotheses MAX** — too many at
   once and we can't tell what worked.
